UI Design

Designing a trustworthy interface for AI-assisted clinical consultations

Design principles

The UI had to feel familiar to NHS users while still making evidence, uncertainty, review state, and account trust visible. We therefore centred the design on familiarity, structure, and trustworthy feedback.

Primary design pressure

Feedback from the sketch stage suggested that users did not want to learn a completely new interface. They preferred layouts that felt close to existing NHS web systems and made common actions obvious.

Core design response

We used structured cards, filters, tabs, and role-specific workspaces so the product felt orderly and easy to scan, while keeping trust cues such as notifications, inline citations, review state, and account-status messaging close to the workflow.

Visibility of system status and trust

Status badges, review banners, AI generating states, unread notification counts, and inline citations are all visible at the point of use, so users can see both what the system is doing and why an answer should be trusted.

Match to clinical workflow

We avoided AI jargon and used terms such as consultation, specialty, urgency, review, assign, request changes, and manual response because they better match how clinicians think and act.

Error prevention and safe control

Structured GP intake, file-size validation, archive and close confirmations, and message-level specialist review actions reduce accidental mistakes. Specialists can approve, approve with comment, request changes, or replace an AI answer manually rather than being forced into one action.

Recognition and efficiency

Citations sit directly under AI answers, queue tabs reduce scanning effort, notification links take users straight to the relevant consultation, and specialists can act on individual AI messages in place.

Consistency, familiarity, and accessibility

Users preferred the sketch direction that looked more like the current NHS website. We therefore leaned into a calm visual language with clear boxes, familiar spacing, repeated status colours, and obvious hierarchy.

Recovery and lightweight help

Not-found states, retry behaviour, resend-verification and password-reset pages, specialist feedback, and the report site together provide enough help without overloading the main consultation screens themselves.

User-centred workflow mapping

Before building the prototype, we mapped the product around the core workflow: authentication, role entry, consultation lists, structured query creation, chat detail, and specialist response. That same spine is now extended with email verification, password reset, profile management, live AI streaming, and a fuller admin governance workspace.

GP

Needs a homepage with tabs, search, and specialty or date filters, a structured consultation form with patient context and attachments, and a chat view where review state and evidence remain visible.

Specialist reviewer

Needs queue and assigned views, fast access to the relevant consultation, clear role labelling, and message-level review controls directly on each AI answer.

Admin

Now includes dashboard analytics, user and chat management, audit logs, and guideline ingestion rather than only passive observability.

Chosen workflow structure

Authentication and recovery Users can register, verify email, resend verification, reset passwords, and then enter the role-appropriate workspace.
→
Homepage and new consultation GPs review consultations through tabs, search, and filters, then create a new consultation with title, patient age, sex, specialty, urgency, notes, question, and optional files.
→
Chat interface and AI response The initial clinical question is sent from the detail page and the AI answer streams into the thread with citations and visible review state.
Automatic submission to review In this workflow, the first GP message moves the consultation into the submitted review workflow instead of relying on a separate manual escalation step.
→
Specialist queue and message review Specialists assign a case to themselves, approve an AI answer, approve with comment, request changes, or replace it with a manual response before closing the consultation.
→
Admin governance layer Admins use a dedicated sidebar workspace for analytics, user and chat controls, audit logs, and guideline uploads.

Workflow nuance

Specialist involvement is modelled as an automatic transition after the first GP message, moving the consultation into the specialist workflow while preserving review controls.

Key early decision

The most important product decision was to keep the GP and specialist sides visually close but operationally distinct. That gave users NHS-like familiarity while still avoiding confusion about who was acting in the system.

Sketches and initial concepts

The design process used two sketch stages. Sketch 1 captured rough ideas quickly, while Sketch 2 made the interface more structured and closer to the look and feel users already associated with NHS web systems.

Sketch 1: rough exploratory layout

The first sketch was intentionally rough. It helped us map page types and content, but it still felt too loose and abstract to communicate a confident NHS-facing workflow.

Sketch 2: structured NHS-like direction

Users preferred Sketch 2 because it looked more like the current NHS website. They highlighted the clearer structure, boxes, icons, and content regions, which made the layout feel easier to navigate and faster to understand.

Main outcomes from sketch review

Users preferred the more NHS-like sketch because it reduced the learning curve. Structured boxes, tabs, and list areas made navigation feel smoother. The chosen direction became the basis for the shipped landing, consultation, and review workflows.

Prototype and iteration

The prototype stage used a Figma model to test authentication, role landing or homepage states, the new-consultation form, and the GP-specialist chat journey. The final product keeps that backbone but extends it with safer account flows, file upload, inline citations, streaming AI responses, and admin governance screens.

What carried into the refined design

The header now explicitly communicates which side of the site the user is on. Notifications are part of the core interaction model so users can react quickly to assignment, specialist messages, approvals, and rejections. The prototype's transparency goal is now implemented as inline citations with document links directly on AI answers. Attachments are no longer just a design direction: PDF and text uploads are supported and contribute file context to generation. The auth flow has expanded with verification, recovery, and profile management to support a production-like multi-role workflow.

Final design

The final UI keeps an NHS-familiar structure while broadening the original GP-specialist prototype. The core consultation journey remains central, with stronger auth flows, live AI streaming, message-level review state, and a full admin governance workspace.

Landing page and auth entry

The public landing page establishes the product as an NHS advice-and-guidance assistant and makes GP and specialist entry points clear. Signed-in users also see session state, while admin entry is surfaced from authenticated admin context rather than as a default public card.

GP consultation flow

The GP side now includes submitted, under-review, and closed tabs, search plus specialty or date filters, archiving, and a structured consultation form with title, age, sex, specialty, urgency, additional clinical context, and optional files. The first clinical question is then sent from the detail view and the consultation moves into submitted review state automatically.

Specialist review flow

The specialist side now has both a queue and a separate My Assigned view. From the detail page, specialists assign a case to themselves, review AI answers individually, approve with comment, request changes, or replace an answer manually, while revised AI output can stream back into the same thread.

Admin workspace

The admin role keeps the same visual language but adds a shared sidebar for dashboard analytics, user management, chat management, audit logs, and guideline upload or ingestion.

Strengths
  • The interface stays close to familiar NHS website patterns, which lowers the learning curve.
  • Structured intake captures patient context before generation and keeps the consultation timeline orderly.
  • Inline citations, per-message review badges, notifications, and loading states reinforce trust and orientation.
  • The admin workspace now supports governance beyond metrics alone through users, chats, audit, and guideline tools.
Trade-offs
  • Live AI generation uses server-sent events with polling fallback rather than a fully push-based model everywhere in the UI.
  • The visual style stays intentionally restrained, prioritising legibility over novelty.
  • The richer GP intake form captures safer patient context, but it is longer than the prototype's minimal version.
  • The admin view is optimised more for desktop governance than for mobile usage.
