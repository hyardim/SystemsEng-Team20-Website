Research

Technical rationale behind the architecture

The project’s research emphasis is architectural and engineering-oriented: how to combine retrieval-augmented generation, clinical evidence grounding, and human specialist oversight into one coherent system that can be deployed, tested, and reasoned about transparently.

Why retrieval-augmented generation?

A generic LLM can provide fluent text, but fluency alone is not enough for clinical support. The implemented design uses retrieval so generation is conditioned on indexed guideline chunks from curated document sources. This improves answer traceability and creates a concrete citation object that can be shown in the user interface.

Problem with plain generation

Unconstrained models can hallucinate recommendations, omit provenance, and blur the difference between evidence and invention.

Reason for RAG

Retrieved chunks become explicit context. The answer can then cite the exact subset of supporting evidence used in generation.

Why a separate RAG service?

Isolation of concerns

Clinical retrieval, embeddings, and generation logic evolve independently from authentication and chat CRUD operations.

Operational flexibility

The model runtime may be local via Ollama or external through a compatible endpoint, so a dedicated service boundary is useful.

Testability

Retrieval modules, prompt construction, citation rules, and generation routing can be tested without entangling frontend logic.

Technology selection rationale

Decision Rationale Trade-off
React + TypeScript SPA Good fit for route protection, multiple dashboards, and strongly typed page-level workflows. Requires client-side coordination for polling and local auth persistence.
FastAPI backend Rapid API development, clean dependency injection, and strong Python ecosystem fit with auth and data tooling. Background-task and startup patch patterns can become less disciplined than migration-first setups.
PostgreSQL + pgvector Keeps operational data and vector retrieval in the same database family while supporting semantic search. Combining transactional and vector concerns can complicate scaling later.
sentence-transformers embeddings Practical local embedding pipeline with an established ecosystem. Embedding quality and speed remain bounded by local hardware and chosen model size.
Polling instead of websockets Simpler deployment model for notifications and in-progress review state changes. Less efficient than server push and introduces deliberate latency.

Clinical safety and grounding decisions

  • Low-relevance retrieval results are filtered before they can influence generation.
  • Boilerplate phrases such as copyright or supplementary material are actively rejected from citation use.
  • Chunks without source-path metadata are excluded to avoid producing citations that cannot be opened.
  • If evidence quality is insufficient, the service returns a no-answer style response rather than inventing a recommendation.
  • Specialist review exists at both chat and per-message level, making clinical oversight part of the normal control flow.

Evolving architecture and research honesty

The repository contains a mixture of mainline runtime code and newer modular RAG logic. This is not unusual in a student engineering project, but it matters for honest reporting. The website therefore treats the architecture as implemented and evolving rather than pretending every documented path is equally productionised.

Examples of evolution

  • There is an older frontend API client alongside a newer service layer.
  • An unused backend health router exists outside the main included routers.
  • Two RAG API paths co-exist: one drives the deployed app while some newer modular logic is covered in tests.
  • Some documentation mentions older model-routing ideas that are not the primary live path.

Research stance

The most important design question was not “which model is strongest?” but “how should model output be constrained, reviewed, and evidenced inside a clinical workflow?”
