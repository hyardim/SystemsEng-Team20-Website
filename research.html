Research

Related systems, evidence, and technical choices

Related Projects Review

We reviewed existing medical AI systems to understand what strong clinical assistants already do well. The most useful patterns were consistent: evidence grounding, structured reasoning, and clear signalling that the system supports clinicians rather than replacing them.

OpenEvidence

OpenEvidence is a medical search tool designed for clinicians, with a stronger focus on showing the evidence behind an answer than on sounding conversational [1][2]. It stood out to us because it helps users get from a clinical question to useful supporting sources quickly. The main takeaway for Ambience was simple: trust is easier to build when citations stay visible in the answer instead of being tucked away as an extra.

Main features

  • Clinician-focused search over medical evidence and guidance rather than open-ended conversation for the general public.
  • Source attribution remains attached to the answer itself, which helps the user judge where the system's claim is coming from.
  • The product positions itself as a professional decision-support tool, not as an autonomous medical advisor.
  • Its "why this source was cited" style of explanation is especially relevant because it makes the retrieval step visible instead of hiding it behind the final answer.

What Ambience learned

  • Citations should be visible at the same level as the answer text.
  • The retrieved evidence should feel like part of the product, not a hidden backend step that the user never sees.
  • A narrow clinician-facing scope makes the system easier to justify and evaluate than a broad "AI doctor" framing.
  • Evidence transparency is not just a UI detail; it directly shapes whether a clinician can trust or challenge the answer.

References

  • [1] OpenEvidence, "About."
  • [2] OpenEvidence, "New Feature: Why Was This Source Cited?"

Glass Health

Glass Health is a useful comparison because it puts structure first. Its clinical Q&A, differential diagnosis, and plan-building workflows show that medical AI is much easier to review when responses are split into clear sections rather than one long answer [3]. For Ambience, that reinforced the need to present evidence in a format that both GPs and specialists can scan and discuss easily.

Main features

  • Structured differential diagnosis and assessment-and-plan generation instead of only free-text responses.
  • Evidence-backed answers with inline numbered citations, which gives the user a clearer reading path through the output.
  • Clinical workflow emphasis rather than a single plain chat box, so the product feels organised around medical tasks.
  • Output structure is part of the value proposition: the system helps users scan, compare, and review information rather than only read it.

What Ambience learned

  • Structured outputs reduce scanning effort for busy clinicians.
  • Reasoning should be grouped into meaningful sections rather than one long paragraph that is harder to review.
  • The UI should make escalation and review feel normal, not exceptional or bolted on afterwards.
  • Clinical support systems benefit from turning model output into a workflow object that can be checked, discussed, and revised.

References

  • [3] Glass Health, "Features."

Med-PaLM 2

Med-PaLM 2 is not a directly comparable deployed product, but it is an important research reference because it frames medical LLM quality in terms of safety, helpfulness, and clinician preference rather than raw model fluency alone [4][5]. It is relevant to Ambience because it shows both the promise and the limits of strong medical language models: high benchmark quality is useful, but it does not remove the need for grounding, oversight, and clear deployment boundaries.

Main features

  • Medical question-answering research with formal human evaluation rather than only automated benchmark scoring.
  • Measures quality across long-form answers, safety, medical consensus, and physician preference.
  • Treats deployment readiness as a separate question from benchmark strength, which is especially important in clinical contexts.
  • Acts as a research reference point for what advanced medical LLM performance can look like under controlled evaluation.

What Ambience learned

  • High-quality answers still need explicit safety boundaries before real use.
  • Capability research should not be confused with autonomous clinical readiness.
  • Model strength alone is not enough; grounding and workflow design still matter for safe use.
  • Human specialist oversight remains the clearest safety layer in our final design, especially when the model output may influence real care decisions.

References

  • [4] Google Research, "Med-PaLM."
  • [5] K. Singhal et al., "Toward expert-level medical question answering with large language models."

Cross-project takeaway

Across all three examples, the strongest pattern was the same: good clinical AI does not just answer questions, it grounds them in evidence, structures them for quick review, and leaves the final judgement with human clinicians.

Technology Review

The final implementation in the sibling Ambience repository uses three services (frontend, backend, and rag_service) plus PostgreSQL, Redis, and optional nginx. The research question was not simply which tools were popular, but which combination created the most grounded, testable, and clinically defensible workflow.

Architecture Choice

The first major design decision was architectural: should Ambience-AI-1.5 behave like a plain medical chatbot, a single monolithic web service, or a system with a dedicated retrieval and generation layer? We chose the third option. In the implemented system, the main backend handles application workflows such as authentication, chats, and review state, while a separate RAG service handles ingestion, retrieval, routing, and grounded answer generation [6].

Options reviewed

  • Plain LLM chatbot: fastest to prototype, but weak on provenance, updateability, and clinical traceability.
  • Single monolithic backend: simpler at first, but it would tightly couple authentication, chat workflows, ingestion, retrieval, and model logic in one service.
  • Dedicated RAG service: adds some operational complexity, but keeps the evidence pipeline separate from the rest of the application.

Final decision

  • Chosen: a separate RAG service behind the main application backend.
  • This was the strongest fit for the project because grounded retrieval is central to trust, safety, and citation quality.
  • It keeps clinical evidence processing separate from account and workflow APIs, which makes the overall system easier to reason about.
  • It also makes ingestion, chunking, retrieval, reranking, and provider routing easier to test and evolve without destabilising the whole application.

References

  • [6] P. Lewis et al., "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks."

Deployment and Inference Path

Ambience-AI-1.5 is a browser-based system, so the relevant deployment comparison was not really mobile versus desktop hardware. The more important question was where inference should happen: only on a local model, only on a cloud-hosted model, or through a hybrid setup that can use both depending on the request. This decision mattered because it affects cost, latency, privacy, and the system's ability to handle more demanding clinical prompts.

Options reviewed

  • Local-only inference: cheapest and easiest for development, with better control over where data is processed, but local hardware limits throughput and model size [7][8].
  • Cloud-only inference: better for large or complex prompts, but introduces ongoing cost, an external dependency, and stronger governance concerns around sending data off-machine.
  • Hybrid local/cloud inference: more complex to implement, but allows the system to keep routine requests local while escalating harder cases when needed.

Final decision

  • Chosen: hybrid routed inference.
  • This was the most balanced option because it avoids paying cloud cost for every query while still keeping a path for more demanding requests.
  • It also fits the project's safety goals better than a single fixed deployment path, because clinically riskier or more ambiguous prompts can be treated differently from routine ones.
  • In the implemented system, the router considers factors such as complexity, prompt size, clinical risk, and retrieval ambiguity before choosing whether to keep inference local or escalate it.
  • That makes the deployment decision part of the system's reasoning strategy, not just an infrastructure detail.

References

  • [7] M42 Health, "Llama3-Med42-70B."
  • [8] Ollama, "OpenAI compatibility."

Retrieval and Ranking Algorithms

The core algorithmic question was how to retrieve clinically relevant evidence reliably enough for grounded answer generation. In this project, a weak retrieval decision would directly reduce citation quality, because the answer can only be as good as the evidence it receives. For that reason, the final system does not rely on one algorithm alone; it combines chunking, retrieval, fusion, filtering, and reranking into one explicit pipeline.

Options reviewed

  • Keyword-only retrieval: useful for exact medical terms, abbreviations, and source-specific phrasing, but too brittle when the user words a question differently from the source document.
  • Vector-only semantic retrieval: better at paraphrase and conceptual similarity through embeddings, but it can surface passages that are semantically related without being the best grounding evidence [9][12].
  • Hybrid retrieval with ranking stages: more complex, but stronger overall because lexical and semantic evidence can complement each other before a final reranking step [10][11].

Final decision

  • Chosen: sentence-aware chunking, vector search, keyword search, reciprocal rank fusion, filtering, cross-encoder reranking, deduplication, and citation assembly.
  • This was the best fit because it improves recall without giving up precision, which is important for clinical support where irrelevant evidence is almost as problematic as missing evidence.
  • The implemented retrieval pipeline is explicit and inspectable, with clear stages for processing, fusion, reranking, and final citation shaping rather than one opaque search step.
  • That makes the algorithmic choice easier to justify on the website as well: the system is not just "using AI", it is using a layered retrieval strategy designed to improve grounding quality.

Languages, Frameworks, and APIs

Once the architecture was fixed, the remaining technology review focused on which languages and frameworks best supported a typed frontend, a testable Python API layer, a retrieval-aware database, provider-flexible inference, and a reproducible multi-service runtime. The goal was not to assemble the most fashionable stack, but to choose technologies that fit the actual work of the project: role-based dashboards, FastAPI services, vector retrieval, and medical-document processing.

Options reviewed

  • Frontend stack: React SPA with Vite versus heavier SSR-first approaches. For this project, most pages are application screens rather than public SEO pages, so a typed single-page frontend was a better fit [13][14].
  • Backend framework: FastAPI versus larger full-stack Python frameworks. FastAPI aligned better with service-style APIs, validation, and async request handling [15].
  • Data and retrieval layer: PostgreSQL with pgvector versus a separate specialist vector database. Keeping relational and vector concerns in one database family simplified the deployment model [12].
  • Inference interface: provider-specific clients versus OpenAI-compatible local/cloud endpoints. Ollama made the local path easier to work with because it exposes a compatible API while keeping model serving relatively simple during development [8].
  • Container runtime: Docker Compose versus managing each service separately in development. For a stack split across frontend, backend, RAG, database, and cache services, Compose made the setup easier to reproduce and run consistently [16].

Final decision

  • Chosen frontend: React 19, TypeScript, and Vite for a dashboard-style SPA with strong typing and fast local iteration.
  • Chosen backend: FastAPI because the project is API-heavy and benefits from explicit schemas, validation, and service-oriented routing.
  • Chosen data layer: PostgreSQL plus pgvector so user data, workflow data, and retrieval data remain in one coherent operational stack.
  • Chosen inference interface: Ollama for local model serving together with OpenAI-compatible local/cloud endpoints, so model providers can be changed with less application-level churn.
  • Chosen runtime tooling: Docker Compose so the frontend, backend, RAG service, database, and supporting services can be started and tested as one reproducible stack [16].
  • Together, these choices support the main priorities of the project: typed user interfaces, modular Python services, grounded retrieval, and deployment that stays practical to run and test.

Technology takeaway

The chosen stack is not the most minimal possible one, but it is the one that best supports grounded answers, modular testing, and explicit safety boundaries for a clinical support system.

Summary of Technical Decisions

Ambience was designed as a clinical support tool, not a general-purpose AI doctor.

The cards below summarise the technical decisions that shaped how the system works:

Grounding first

Answers are produced from retrieved guideline material and returned with explicit citations, so the system is anchored to evidence rather than unsupported model free text.

Separate RAG service

Ingestion, retrieval, reranking, routing, and generation live outside the main backend so that the evidence pipeline can be tested and evolved without destabilising the rest of the application.

Hybrid inference path

Local Med42 is viable for routine development and lower-cost requests, while harder, longer, or riskier prompts can be escalated to a stronger cloud path when necessary.

Hybrid retrieval pipeline

Keyword search and vector search are fused, reranked, deduplicated, and converted into citation objects before the answer is shown to the user, improving both recall and final evidence quality.

These four points capture the highest-level design stance of the project. The remaining technology choices then support that stance at implementation level, turning the system into a role-aware clinical support platform rather than a generic LLM interface.

  • The frontend uses React, TypeScript, and Vite because the product is primarily a role-based application with dashboards, queues, and typed interaction flows rather than a content site.
  • The backend uses FastAPI and SQLAlchemy because the project is API-heavy and benefits from explicit validation, clear schema handling, and close alignment with the wider Python retrieval stack.
  • PostgreSQL with pgvector was chosen so operational data and vector retrieval data could remain in one coherent deployment model instead of being split across unrelated systems.
  • The RAG service uses sentence-transformers, cross-encoder reranking, and Redis-backed retry handling because evidence quality and robustness matter more here than keeping the pipeline minimal.
  • OpenAI-compatible local and cloud providers were chosen to keep deployment flexible, making it possible to balance capability, cost, and infrastructure control.
  • Most importantly, specialist review remains part of the normal workflow, which makes the system a clinical support tool with human accountability rather than an autonomous decision-maker.

This website intentionally describes implementation details, deployment paths, safeguards, testing, and known limitations rather than product marketing claims.
