Development Blog

Weekly progression log for Ambience-AI-1.5

Weeks 23–24

In these two weeks, we focused on closing everything properly. We ran one more full testing pass, fixed the final bugs we could find, and made sure no major issues were left open before handover.

We also spent a lot of time finishing the report website and checking that the write-up matched the real system. It was mostly polish work, but it mattered because this was the final version.

Weeks 21–22

In these two weeks, we focused on performance and reliability. We tested how the system behaved under heavier use, including response times, error handling, and fallback paths.

As expected, this found a few rough edges. We fixed those issues, reran tests, and also cleaned up accessibility and UI problems that had been hanging around.

Weeks 19–20

In these two weeks, we started formal testing. We wrote and ran end-to-end scenarios across GP, specialist, and admin flows, and logged every issue we found.

We then moved into a steady fix-and-retest routine. There were more bugs than we hoped for, but nothing that blocked the project.

Weeks 17–18

In these two weeks, we prepared for deeper testing by making the full stack behave more consistently. We fixed integration issues between frontend, backend, and retrieval.

We also fixed awkward edge cases in the specialist review flow and improved our container setup so everyone could run the same environment more reliably.

Weeks 15–16

In these two weeks, we focused on practical features now that the core consultation path was working. We added structured forms, file attachments, notifications, and basic admin tooling.

This made a big difference to day-to-day usability. The system started to feel like a real product, not just a demo flow.

Weeks 13–14

In these two weeks, we reviewed early prototypes and saw that the interaction design was getting too complicated. We had separate paths that were hard to maintain and not very clear.

We merged them into one cleaner consultation flow and tightened the specialist review step. It was a lot of rework, but the result was easier to follow.

Weeks 11–12

In these two weeks, we worked on recovery after the server setback. We rebuilt a working local-first setup, reconnected inference and retrieval, and checked that fallback behavior still worked.

We were careful to keep the GP and specialist experience the same while changing the infrastructure. That helped us get back on track quickly.

Weeks 9–10

In these two weeks, we hit the hardest point in the project. We lost access to the original Gaudi 2 server we had been building on, which forced us to rethink our deployment plan.

Instead of chasing another identical setup, we switched to a local-first approach with cloud fallback. It was a stressful pivot, but it kept the project moving.

Weeks 7–8

In these two weeks, we worked on stabilizing what we had on the server. Early runs worked, but setup was still fragile and hard to repeat across the team.

We documented the setup properly, improved reproducibility, cleaned up citation mapping, and refined the specialist review flow based on early feedback.

Weeks 5–6

Most of these two weeks were setup work. We configured the Gaudi 2 environment, fixed driver/runtime compatibility issues, and worked through repeated failures.

By the end of week 6, we had Med42 running reliably with early retrieval connected. That gave us a stable base to build on.

Weeks 3–4

In these two weeks, we turned interview findings into clear project requirements and prioritized them using MoSCoW. This helped us decide what had to be delivered first.

The top priorities were grounded retrieval, specialist approval, and safe fallback behavior. We also confirmed Gaudi 2 as the deployment direction at that point.

Weeks 1–2

In the first two weeks, we spoke to specialists in rheumatology and neurology to understand the real Advice and Guidance workflow and what they actually needed from the system.

From those conversations, we set the initial scope around a GP-to-specialist process, early model direction, and a focused evidence base for the first version.
