Development Blog

Bi-weekly progression log for Ambience-AI-1.5

Weeks 23–24

This final period focused on packaging and handover. We consolidated evidence for the report website, documented how to run each service, and made sure the system could be demonstrated end-to-end without manual setup steps.

We also ran one last pass of regression testing across the full workflow and fixed the small defects that still affected reliability or usability. At this point we deliberately avoided feature work and concentrated on making what we already had feel complete.

By the end of Week 24, we had a stable demonstration, a clearer handover story, and supporting documentation that matched what we actually built and tested.

Weeks 21–22

These two weeks focused on reliability and performance, particularly around retrieval and model routing. We measured typical end-to-end latency, looked for bottlenecks in ingestion and search, and tuned defaults so the system stayed responsive on local hardware.

We also hardened failure behaviour so the user experience stayed consistent even when external services were unavailable. This included better timeouts and clearer error states, and a repeatable way to fall back to the cloud model path when the local route was not sufficient.

Weeks 19–20

This phase was about integration and test coverage. With the local model and cloud endpoint strategy in place, we focused on making the full GP-to-specialist workflow work consistently from submission through to review and final output.

We wrote and ran end-to-end scenarios across roles, then used the defects found to drive a fix-and-retest loop. We also validated edge cases like partial submissions, attachment handling, and review state transitions to reduce surprises later.

Weeks 17–18

This period focused on turning the routing idea into a usable implementation. We integrated the local Med42 8B path as the default and established a cloud fallback route for harder queries, while keeping the API provider-agnostic.

In parallel, we continued improving retrieval quality and citation mapping so the specialist review remained grounded in evidence. We treated this as core safety work and made sure the review UI surfaced what the model used and why.

Weeks 15–16

After lots of extensive with local models, we settled on the same Med42 model, however a less powerful and more lightweight version (8B parameters, compared to the previous 70B) that is able to run locally - in our case on our laptops. The model has also been quantized to 8 bits to reduce hardware requirements further making this run very reasonably locally, with model queries on average returning in under 10 seconds.

For cloud deployment, we currently chose to host via RunPod due to its relatively low cost and ability to avoid charges when the model is not actively running. However, the system was designed with a non-provider-specific endpoint, making it easy to switch between compatible cloud hosting solutions in future if requirements or infrastructure change.

Weeks 13–14

This week, Intel informed us that unfortunately in a few weeks time they would need to reclaim the server we were using as the Gaudi's contract was expiring. This made us rethink our strategy for the backend to run the model, and we concluded that we wouldn't be able to use the Med42 70B parameter version as it was much too demanding to run locally, and other cloud solutions for the uptime we were expecting were expensive. We finalised a plan to implement an intelligent routing layer that would allow us to run a more lightweight model locally, and for more demanding queries, we would outsource to a cloud server that would run the 70B parameter version.

We have also got the RAG fully working, though in need of much testing and optimisation. Improvements to performance will likely come after lots of trial and error in this field.

Weeks 11–12

Requirement refinement with our clients and prototype review led to a major product decision: we decided to merge the initially planned GP query system and separate chatbot into one,where generation, review and chat history stay connected.

This was an important simplification for us and the end user. We removed the split system from the frontend and also improved traceability so review decisions were easier to follow and audit.

Work continued steadily on the backend RAG system, with our ingestion and chunking stage complete and the frontend design was also slightly tweaked in line with client feedback.

Weeks 9–10

This week we begin working on the software after finishing planning, and making early design decisions. Up to now, most of our progress has been about understanding the workflow, reducing risk, and deciding what needs to be built first.

The most important part of our software for answer generation is the RAG system, so we focused on that first, starting with the ingestion phase which splits up the guideline documents into chunks to be referenced later. We tried different strategies, and in the end settled chunks being created primarily by document section/headings, then split into sentence-based chunks. We also started to set up the database schema and vector DB.

There was also significant work on the frontend, with a prototype being created to show the specialists our first draft UI.

Weeks 7–8

We spent these two weeks improving robustness on the Gaudi 2 path and aligning implementation details with the agreed requirements.

Since the Med42 70B was now running correctly, we focused on making the setup repeatable and less fragile. We cleaned up configuration steps, re-checked drivers and the kernel after updates and documented the server setup progress.

Most of our effort goes into consistency and safety. We tighten citation mapping, refine specialist review behavior through feedback and document deployment friction points as we go.

Weeks 5–6

A significant part of this period was spent setting up and validating the original Gaudi 2 server environment. We treated this as core engineering work because model runtime behavior and infrastructure constraints directly affected design.

A major milestone this week was installing the Med42 70B model on the server and getting stable responses. That process took longer than expected as we had to work through low-level setup tasks first, including driver updates, runtime compatibility checks and environment configuration before the model would run on the unique and powerful HPUs.

Weeks 3–4

We wrote up what we learnt in the interview into explicit functional and non-functional requirements, then prioritized them with MoSCoW. We also identified key risk areas that we would need to address: uncertain model outputs/hallucinations, deployment uncertainty, and governance expectations. We finished off the fortnight by planning the initial deployment path around the medically trained MED42 LLM on Intel's powerful AI specialised Gaudi 2 server.

Weeks 1–2

We started by mapping the Advice and Guidance pain points with specialist input from Rheumatology and Neurology. The main requirement was a structured GP request that could lead to a specialist-checked response rather than an unreviewed AI output.

During these first two weeks, we narrowed scope to neurology and rheumatology guidance so retrieval could be grounded in a realistic evidence base. We also agreed that evidence needed to stay visible so the specialist could review the response before anything was officially submitted to a GP.

That gave us the first version of the requirements baseline: keep answers grounded in guideline evidence and preserve specialist oversight, with the response easy to review before anything was officially submitted to a GP.
