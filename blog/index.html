Development Blog

Bi-weekly progression log for Ambience-AI-1.5

Weeks 23–24

In the first two weeks, we spoke to specialists in rheumatology and neurology to understand the real Advice and Guidance workflow and what they actually needed from the system. The main requirement was a structured GP request that could lead to a specialist-checked response rather than an unreviewed AI output.

From those conversations, we set the initial scope around a GP-to-specialist process, clear evidence visibility, and a focused neurology and rheumatology knowledge base for the first version.

That gave us the first version of the requirements baseline: keep answers grounded in guideline evidence, preserve specialist oversight, and make the response easy to review before anything reached a GP. evidence for the report website and closed remaining defects. By the end of Week 24, we had a cleaner handover story that matched what we actually built and tested.

Weeks 21–22

This period focused on non-functional validation and repeated bug-fix loops. Testing was intentionally iterative so we could re-run critical scenarios after each fix batch.

We checked latency on routine cases, stress-tested failure and recovery paths, and refined accessibility and workflow clarity in core screens. Each fix was followed by another targeted run to confirm that we were improving without introducing regressions.

Weeks 19–20

The final stretch deliberately prioritized testing as a major body of work. We started with end-to-end functional checks and integration scenarios across role-based workflows.

We ran representative GP, specialist and admin scenarios end-to-end, including low-confidence refusal paths and fallback behavior. The defects found in this wave fed directly into an ongoing fix-and-retest loop.

Weeks 17–18

Before entering final testing, we focused on reducing integration risk between frontend, backend and retrieval services.

We concentrated on edge cases in review-state transitions, reliability around retrieval and fallback routing, and a repeatable container-based setup. The goal was to enter the final testing stretch with fewer integration surprises.

Weeks 15–16

We used this phase to complete the features that make the system operationally usable rather than a narrow demo.

During these weeks, we added the surrounding platform capabilities that clinicians and administrators would actually need in practice: structured submission, optional attachments, notification handling and admin-level observability.

Weeks 13–14

Requirement refinement and prototype review led to a major product decision: merge assistant-like interactions into one consultation-based journey where generation, review and decision history stay connected.

This was an important simplification for us. We removed split interaction paths, enforced the specialist gate before GP-visible completion and improved traceability so review decisions were easier to follow and audit.

Weeks 11–12

By this point, the team had a working local-first path and resumed steady feature work. The key objective was to avoid user-facing workflow disruption despite backend routing and environment changes.

We re-established stable inference and retrieval behavior, then validated fallback behavior under failure scenarios. Importantly, we kept the consultation and specialist review journey consistent so the infrastructure pivot did not create a different user experience.

Weeks 9–10

This week, Intel informed us that in a few weeks time they would need to reclaim the server we were using as the Gaudi's contract was expiring. This made us replan our strategy for the backend to run the model, and we concluded that we wouldn't be able to use the Med42 70B paramater version as it was much too demanding to run locally, and other cloud solutions for the uptime we were expecting were expensive. We finalised a plan to implement an intelligent routing layer that would allow us to run a more lightweight model locally, and for more demanding queries, we would outsource to a cloud server that would run the 70B parameter version.

Weeks 7–8

We are spending these two weeks improving robustness on the Gaudi 2 path and aligning implementation details with the agreed requirements.

The first Med42 70B runs are succeeding, so we are focusing on making the setup repeatable and less fragile. We clean up configuration steps, re-check drivers after updates and document the server setup decisions so the team can reproduce the working state.

Most of our effort goes into consistency and safety. We tighten citation mapping, refine specialist review behavior through feedback and document deployment friction points as we go.

Weeks 5–6

A significant part of this period was spent setting up and validating the original Gaudi 2 server environment. We treated this as core engineering work, not overhead, because model runtime behavior and infrastructure constraints directly affected design.

A major milestone in this phase was installing the Med42 70B model on the server and getting a stable first run. That process took longer than expected because we had to work through low-level setup tasks first, including driver updates, runtime compatibility checks and environment configuration before inference would run reliably.

By the end of Week 6, we had environment checks, early retrieval and first-generation flow running in controlled scenarios. This gave us enough confidence to continue implementation on top of a live setup instead of only mock components.

Weeks 3–4

We converted interview findings into explicit functional and non-functional requirements, then prioritized them with MoSCoW. At this stage, we also identified key risk areas: uncertain model outputs, deployment uncertainty, and governance expectations.

We used this period to lock the must-have baseline around grounded retrieval, specialist approval and safe fallback behavior. In parallel, we drafted measurable acceptance criteria for latency, auditability and reliability, and planned the initial deployment path around Intel's Gaudi 2 server environment available at that time.

Weeks 1–2

We started by mapping the Advice and Guidance pain points with specialist input from Rheumatology and Neurology. The main requirement was a structured GP request that could lead to a specialist-checked response rather than an unreviewed AI output.

During these first two weeks, we narrowed scope to neurology and rheumatology guidance so retrieval could be grounded in a realistic evidence base. We also agreed that evidence needed to stay visible so the specialist could review the response before anything was officially submitted to a GP.

That gave us the first version of the requirements baseline: keep answers grounded in guideline evidence and preserve specialist oversight, with the response easy to review before anything was officially submitted to a GP.
