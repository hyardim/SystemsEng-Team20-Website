Overview

The cache layer uses scoped keys and cache-aside reads so hot query results can be served from Redis instead of repeatedly hitting the database. Writes trigger targeted invalidation across GP, specialist, and admin views, while a bounded in-process fallback keeps the application functional if Redis is unavailable. This keeps cache behavior predictable even during partial outages.

This matters for both performance and correctness: users get faster page loads, backend query pressure stays lower, and explicit invalidation rules reduce stale-data surprises after writes. The user-scoped key strategy also prevents cross-tenant leakage.

Architecture

The cache layer uses Redis as primary storage with an in-process Python dict as a fallback when Redis is unavailable. All cache keys are scoped by user_id and resource type to prevent cross-user data leakage:

# backend/src/utils/cache.py (simplified)
class Cache:
    def _scoped_key(self, key: str, user_id: int, resource: str) -> str:
        return f"{resource}:{user_id}:{key}"

    def get_sync(self, key, user_id, resource) -> Any | None:
        scoped = self._scoped_key(key, user_id, resource)
        try:
            raw = self._redis.get(scoped)
            return pickle.loads(raw) if raw else None
        except RedisError:
            entry = self._local.get(scoped)
            if entry and entry[1] > time.time():   # check TTL
                return entry[0]
            return None

    def set_sync(self, key, value, ttl, user_id, resource) -> None:
        scoped = self._scoped_key(key, user_id, resource)
        try:
            self._redis.setex(scoped, ttl, pickle.dumps(value))
        except RedisError:
            self._local[scoped] = (value, time.time() + ttl)

    def delete_pattern_sync(self, pattern, user_id, resource) -> None:
        full_pattern = f"{resource}:{user_id}:{pattern}*"
        try:
            keys = self._redis.keys(full_pattern)
            if keys:
                self._redis.delete(*keys)
        except RedisError:
            prefix = f"{resource}:{user_id}:"
            self._local = {k: v for k, v in self._local.items()
                           if not k.startswith(prefix)}

cache = Cache()

Read/write methods share the same scoped-key format, which makes cache invalidation predictable. The fallback dictionary is intentionally best-effort: it preserves service continuity during Redis outages, but long-term coherence still depends on Redis availability.

Cache-Aside Pattern

The standard read path is: check cache → miss → query DB → store result:

# backend/src/services/chat_service.py
def get_chat_list(db, user, skip, limit, status_filter) -> ChatListResponse:
    cache_key = f"list:{skip}:{limit}:{status_filter}"

    cached = cache.get_sync(cache_key, user_id=user.id, resource="chat_list")
    if cached:
        return cached

    chats  = chat_repository.list_for_user(db, user.id, skip=skip, limit=limit,
                                            status=status_filter)
    result = ChatListResponse(chats=[chat_to_response(c) for c in chats])

    cache.set_sync(cache_key, result, ttl=300,   # 5-minute TTL
                   user_id=user.id, resource="chat_list")
    return result

Cache-aside keeps write paths simple because only read paths interact with cache population. The tradeoff is bounded staleness up to TTL, which is why mutation handlers pair this pattern with explicit invalidation calls.

Cache Invalidation

After any mutation, all affected cache entries are invalidated by pattern. For example, after a specialist submits a review:

# backend/src/services/specialist_review.py
def _invalidate_chat_views(chat: Chat, specialist_id: int | None) -> None:
    # Invalidate GP's view of this chat and their chat list
    cache.delete_pattern_sync(cache_keys.chat_detail_pattern(chat.id),
                               user_id=chat.user_id, resource="chat_detail")
    cache.delete_pattern_sync(cache_keys.chat_list_pattern(chat.user_id),
                               user_id=chat.user_id, resource="chat_list")
    # Invalidate specialist's queue/assigned views
    if specialist_id:
        cache.delete_pattern_sync(cache_keys.specialist_queue_pattern(chat.specialty),
                                   user_id=specialist_id, resource="specialist_queue")
    # Invalidate admin caches
    _invalidate_admin_chat_caches(chat.id)
    _invalidate_admin_stats_cache()

The invalidation list mirrors the user roles that can observe the same entity. That fan-out is critical: if only GP caches were cleared, specialist/admin views could continue showing stale status after review actions.

Async-to-Sync Redis Bridge

The underlying Redis implementation is async, while much of the backend service layer uses synchronous handlers. Calling await is not possible in those sync functions, and naive asyncio.run() usage is unsafe when a request loop is already active.

The cache layer resolves this using a dedicated daemon-thread event loop. Sync wrappers submit coroutines to that loop with asyncio.run_coroutine_threadsafe() and block on bounded futures.

# backend/src/utils/cache.py
def _run_sync(coro):
    loop = _get_sync_loop()
    future = asyncio.run_coroutine_threadsafe(coro, loop)
    return future.result(timeout=5)  # timeout guards request handlers from hanging

The background loop is lazily created on first use and reused for process lifetime. A clean shutdown hook stops the loop on exit so the daemon thread does not leak resources during controlled termination.

This pattern keeps sync service code ergonomic while preserving async Redis I/O behavior and bounded failure handling.

CacheKeys Namespace Design

Cache keys follow a structured namespace containing application prefix, resource domain, and user scoping where relevant. The goal is both isolation and precise invalidation.

User-scoped keys enforce cross-tenant safety; wildcard patterns can then target only one user or intentionally fan out across users for shared resources such as a specific chat viewed by GP, specialist, and admin roles.

Design goalKey strategyOperational benefit
Cross-tenant isolationEmbed user:{id} in user-scoped keys.Pattern deletes cannot leak across users.
Targeted fan-out invalidationUse wildcard middle segments for entity views (e.g. one chat across roles).Clears all affected role caches with one operation.
Production-safe scanningUse Redis SCAN/scan_iter patterns.Avoids blocking-keyspace operations under load.

The invalidation path uses scan-based iteration instead of blocking KEYS sweeps, making wildcard cache maintenance safe under production load.

Cross-tenant isolation means a GP's chat list cache must never be readable or modifiable by another user's session. The user:{id} embedding in every key is the primary enforcement mechanism: a pattern delete scoped to chat_list:42:* cannot accidentally match or clear entries under chat_list:43:*. Without this scoping, a wildcard invalidation after a write could clear cached data for unrelated users, causing unnecessary database load, or worse, could serve cached data to the wrong user if key collisions were possible. Targeted fan-out means that when a specialist updates a single chat, only the specific cache keys affected by that entity need to be evicted: the GP's view of that chat, the GP's chat list, the specialist's queue, and the admin overview. This avoids a broadcast invalidation that would wipe the entire namespace and force every concurrent user to re-query the database simultaneously. Production-safe scanning uses Redis SCAN with a pattern rather than the blocking KEYS * command. On a keyspace of any significant size, KEYS * blocks the Redis server for the entire duration of the scan, blocking every other client. SCAN iterates incrementally and yields control between steps, keeping eviction operations safe to run against a live server under load.

Cache TTL values represent a deliberate consistency trade-off. A 5-minute TTL on chat lists means a GP may see a list that is up to 5 minutes stale if an explicit invalidation was somehow missed — for example, during a Redis outage when the fallback dict is active. That bounded staleness is acceptable because the GP's own mutations always trigger immediate invalidation before the response is returned, so their own actions are always reflected correctly. The TTL functions as a safety net that prevents indefinitely stale data in edge cases while keeping the cache effective for repeated reads. Shorter TTLs would increase database query frequency without meaningfully improving consistency for non-mutating users, while longer TTLs would increase the risk window after write operations.
