Overview

File handling is treated as a controlled ingestion path: uploads are validated for policy and size, filenames are sanitized, storage locations are scoped by chat, and metadata is persisted for traceability. Text extraction then feeds relevant file content into RAG context construction with explicit size caps so documents cannot overwhelm the prompt budget. Each step is ordered to block unsafe files before any disk write and to keep database pointers consistent with storage state.

This is important because attachments can improve answer quality, but they also introduce security and reliability risks. Structured validation and bounded extraction reduce those risks while still letting clinicians include supporting material and keep evidence context visible in the final response.

File Validation and Storage

# backend/src/services/chat_uploads.py (simplified)
ALLOWED_EXTENSIONS = {".pdf", ".txt", ".md", ".rtf", ".doc", ".docx",
                      ".csv", ".json", ".xml"}
MAX_FILE_SIZE_BYTES = 3 * 1024 * 1024   # 3 MB

async def upload_chat_file(db, user, chat_id, upload_file) -> FileAttachment:
    # 1. Policy check — is this user allowed to upload to this chat?
    chat = _assert_can_upload(db, chat_id, user)

    # 2. Validate file extension
    ext = Path(upload_file.filename).suffix.lower()
    if ext not in ALLOWED_EXTENSIONS:
        raise HTTPException(400, f"File type '{ext}' is not allowed")

    # 3. Read and check file size
    data = await upload_file.read()
    if len(data) > MAX_FILE_SIZE_BYTES:
        raise HTTPException(413, "File exceeds 3 MB limit")

    # 4. Sanitise filename — strip non-ASCII and dangerous characters
    safe_name = sanitise_filename(upload_file.filename)

    # 5. Write to disk (organised by chat_id subdirectory)
    dest = UPLOAD_DIR / str(chat_id) / safe_name
    dest.parent.mkdir(parents=True, exist_ok=True)
    dest.write_bytes(data)

    # 6. Persist metadata to database
    attachment = FileAttachment(
        chat_id=chat_id,
        filename=safe_name,
        file_path=str(dest),
        file_type=ext,
        file_size=len(data),
        uploaded_by=user.id,
    )
    db.add(attachment)
    db.commit()
    db.refresh(attachment)
    return attachment

Validation order is intentional: authorization first, then format/size checks, then filesystem write, then metadata commit. This avoids writing disk artifacts for requests that would fail policy checks.

Persisting file metadata after successful write ensures database pointers never reference files that failed to save.

Content validation (magic-byte check plus binary heuristic) is deliberately separate from extension validation because extension is user-controlled and trivially spoofed. A file named report.pdf passes extension validation regardless of what is actually inside it. The magic-byte check verifies that the first bytes of the file match the expected binary signature for the declared format — for PDFs this is the %PDF- header. However, magic bytes alone are not sufficient: an encrypted or password-protected PDF has a valid %PDF- header but its body is entirely binary ciphertext. The binary heuristic catches this class of file by measuring the proportion of non-printable bytes in the file content; files exceeding the threshold are rejected before they reach text extraction. Without the heuristic, pdfplumber would attempt to parse the encrypted body, producing either empty output, garbled text, or an unrecoverable exception that could hang the upload thread while holding a database connection open.

File Content in RAG Context

When generating an AI response, uploaded files are extracted and injected into the prompt as an "UPLOADED DOCUMENTS" section:

# backend/src/services/rag_context.py (simplified)
def extract_text(file_path: str, file_type: str) -> str | None:
    if file_type == ".pdf":
        try:
            import pdfplumber
            with pdfplumber.open(file_path) as pdf:
                return "\n".join(page.extract_text() or "" for page in pdf.pages)
        except Exception:
            return None   # gracefully handle corrupted or password-protected PDFs
    elif file_type in (".txt", ".md", ".csv"):
        return Path(file_path).read_text(errors="replace")
    return None

def build_file_context(chat: Chat, extract_text_fn) -> str | None:
    texts = []
    for f in (chat.files or []):
        text = extract_text_fn(f.file_path, f.file_type)
        if text and text.strip():
            texts.append(f"[{f.filename}]\n{text.strip()}")
    return "\n\n---\n\n".join(texts) if texts else None

The extracted text is capped and passed to the RAG prompt builder. If it exceeds the token budget, a was_truncated=True flag is set and a warning banner is shown to the user.

File Context Truncation & Propagation

When extracted text exceeds FILE_CONTEXT_CHAR_LIMIT, truncation is sentence-aware rather than a blunt character cut. The builder tries to trim at the last sentence boundary inside the budget; if none exists, it falls back to the last word boundary.

A visible truncation marker is appended so downstream prompt construction makes truncation explicit to the model rather than silently clipping content.

The result object includes both final file_context and was_truncated. That flag is propagated end-to-end so the user is warned when only part of a document was used during answer generation.

Context build build_file_context_result() returns content plus was_truncated.
Generation payload Backend forwards file_context_truncated=true to generation path.
SSE completion event complete event includes truncation flag.
Frontend reaction onFileContextTruncated() toggles warning banner in query detail view.

PDF extraction also has a fallback path: if extraction returns minimal text (for example image-only or protected PDFs), the service emits a warning string instead of silently passing empty context, e.g. [warning] This PDF appears to be empty, password-protected, or contains only images..

Truncation matters because the RAG prompt has a fixed token budget shared between retrieved guideline chunks, patient context, conversation history, and file content. If a large document is injected in full it can crowd out the guideline evidence that grounds the AI response, degrading answer quality. Sentence-aware truncation keeps as much of the document as possible within the budget while preserving complete clinical statements rather than cutting mid-sentence. The was_truncated flag exists because silent truncation would be clinically misleading: the GP uploaded the document expecting the AI to have read all of it. When was_truncated=True is propagated through the SSE complete event to the frontend, the GP's query detail view shows a visible warning banner indicating that only part of the file was used during answer generation. This allows the GP to make an informed judgment about whether the truncated section contained information relevant to their query before acting on the response.
