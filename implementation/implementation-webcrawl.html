Auto-Sync & Web Crawl

Section 12 of 12 · Back to Implementation

Overview

The auto-sync feature (merged via PR #90, branch feature/guideline-web-sync) automatically discovers, downloads, and ingests clinical guidelines from the web, keeping the RAG index up to date without manual admin intervention. It runs on a configurable schedule and feeds the same 8-stage ingestion pipeline used for manual PDF uploads.

The feature is disabled by default (GUIDELINE_SYNC_ENABLED=false) and requires explicit opt-in. When enabled, it runs on startup and then on a fixed interval. The three components — discovery client, change-detection sync service, and asyncio scheduler — are each independently replaceable.

Source Registry

The set of sources to crawl is defined as a hardcoded registry in web_sources.py. Each entry maps a key to a publisher, specialty, and listing URL.

Hardcoding the registry rather than storing it in the database keeps the crawl configuration under version control. Adding a new source requires a code change and a deliberate deployment rather than a runtime database edit, which reduces the risk of accidentally pointing the crawler at an unintended domain.

Discovery

SourceDiscoveryClient.discover_source() in web_sources.py is responsible for finding all PDF URLs for a given source. The first action for every source is a robots.txt check — if the configured user-agent (AmbienceRAGGuidelineSync/1.0) is disallowed, the source is skipped entirely before any further network requests are made.

For NICE sources, discovery runs a multi-depth crawl:

  1. Fetch the category listing page (e.g. nice.org.uk/guidance/conditions-and-diseases/musculoskeletal-conditions)
  2. Follow subcategory links up to depth 2, visiting up to 120 pages per source
  3. Extract links matching the NICE guideline path regex (/guidance/(ng|cg|qs|ta|...)NNN)
  4. Visit each matched guideline page and extract PDF download links

For BSR sources, the crawler extracts links whose URL or anchor text contains keywords such as guideline, recommendation, or arthritis, without the multi-depth subcategory traversal.

After locating each PDF URL, the client sends a HEAD request to collect metadata — ETag, Last-Modified, and Content-Length — without downloading the file. This metadata is used in the next stage to determine whether the document has changed since the last sync. The depth cap (2) and page limit (120) exist to bound network cost; clinical guideline sites tend to have shallow, well-structured hierarchies so these limits are rarely hit in practice.

Change Detection

GuidelineWebSync.sync_once() in web_sync.py maintains a persistent JSON state file at data/raw/_sync_state/guideline_sync_state.json. On each run it compares discovered documents against this state to decide what needs to be downloaded.

The decision logic per document is:

  1. No prior record → download and ingest
  2. ETag matches stored value → skip (unchanged)
  3. Last-Modified and Content-Length both match → skip
  4. Otherwise → re-download and re-ingest

As a final guard, after downloading the file a SHA-256 content hash is computed and compared to the stored hash. If the hash is identical the ingestion step is skipped even if HTTP headers changed — this handles CDN or server configurations that vary ETag without changing file content. ETag is preferred as the primary signal because it is set by the origin server specifically to indicate content identity; Last-Modified + Content-Length are used as a fallback for servers that do not send ETags.

Downloads stream directly to a .tmp file that is atomically renamed on completion. This prevents a partially downloaded file from being passed to the ingestion pipeline if the connection drops mid-transfer. After processing all discovered documents, any entry in state that was not seen in the current crawl is marked stale: true but is not automatically removed from the pgvector index — stale marking flags the document for manual review rather than silently deleting clinical content.

Ingestion Bridge

After a file is downloaded, _run_ingestion() is called with the destination path and source name. This calls the existing run_ingestion() pipeline — the same EXTRACT → CLEAN → SECTION → TABLE → METADATA → CHUNK → EMBED → STORE sequence used for manual PDF uploads via the admin panel. The web crawl feature adds no new ingestion logic; it only automates the document acquisition step that a human admin would otherwise perform manually.

This design means that any improvements made to the core ingestion pipeline (chunking strategy, embedding model, metadata extraction) automatically apply to web-synced documents without changes to the crawl layer.

Scheduler

GuidelineSyncScheduler in web_scheduler.py is an asyncio-based periodic task wrapper around the sync service. It uses asyncio.wait_for() to enforce a per-run timeout so that a slow or unresponsive upstream site cannot block the scheduler indefinitely. Concurrent runs are prevented via asyncio.Lock() — if a sync is already in progress and a second trigger arrives (e.g. a manual API call during a scheduled run), a SyncAlreadyRunningError is raised, which the API layer translates into a 409 response.

The scheduler fires once on startup (if GUIDELINE_SYNC_RUN_ON_STARTUP=true) and then on a fixed interval. The startup run ensures that any documents published since the last deployment are ingested immediately rather than waiting for the first scheduled interval.

Configuration

All sync behaviour is controlled via environment variables. The feature is off by default — setting GUIDELINE_SYNC_ENABLED=false in a deployment environment means no crawl requests are ever sent, which is appropriate for development and CI environments.

VariableDefaultPurpose
GUIDELINE_SYNC_ENABLEDfalseMust be set to true to activate the scheduler and crawl.
GUIDELINE_SYNC_INTERVAL_MINUTES10080 (weekly)How often the scheduler fires a sync run.
GUIDELINE_SYNC_RUN_ON_STARTUPtrueWhether to run a sync immediately when the RAG service starts.
GUIDELINE_SYNC_TIMEOUT_SECONDS900Maximum wall-clock time allowed for a single sync run before it is cancelled.

The interval default was changed from 720 minutes (12 hours) to 10080 minutes (weekly) in a follow-up commit, reflecting the observation that clinical guidelines are not published at high frequency and a weekly check is sufficient to keep the index current without generating unnecessary crawl traffic.

API Endpoints

Two routes are registered on the RAG service to expose sync control and visibility without requiring shell or container access:

# rag_service/src/api/routes.py

POST /guidelines/sync
# Triggers an immediate sync run.
# Optional body: { source_names: [...], dry_run: bool }

GET /guidelines/sync/status
# Returns current sync state:
# { running, enabled, last_started_at, last_finished_at, last_error, last_result }

POST /guidelines/sync accepts an optional source_names filter so a single source can be re-synced without running the full crawl. The dry_run flag triggers discovery and HEAD-checks without downloading or ingesting — useful for estimating how many documents would be affected before committing to a full run.

GET /guidelines/sync/status exposes the scheduler's live state. The running flag allows the admin UI to show whether a sync is currently in progress. last_result includes a per-source breakdown of documents downloaded, skipped, and failed on the most recent run, giving operational visibility without requiring log access.

CLI Command

A CLI entry point is provided for manual invocation outside of the scheduler:

python -m src.ingestion.cli sync-web \
  --source NICE \
  --dry-run        # discover + evaluate without downloading

The --source flag restricts the run to a named publisher. Without it, all registered sources are crawled. The CLI is primarily used for one-off investigations and for verifying the crawler configuration in a development environment where the scheduler is not running.

Dry Run Mode

In dry-run mode the crawler still fetches listing and category pages and sends HEAD requests to PDFs to evaluate their change status, but skips the actual download and ingestion steps. The output is a count of documents that would be downloaded (new or changed) versus skipped (unchanged), broken down by source.

Dry run is useful before a first sync on a new deployment to confirm that the crawler can reach all configured sources and that the expected number of documents are discoverable. It is also used after adding a new source to the registry to validate the crawl depth and link extraction logic before any files are written to disk.

Key Design Decisions

robots.txt compliance. The crawler checks robots.txt before any page fetch and identifies itself with a real user-agent string (AmbienceRAGGuidelineSync/1.0). This is not just a courtesy — NICE and BSR are authoritative clinical publishers whose continued availability depends on not being overloaded by automated clients, and compliant crawling reduces the risk of IP blocks.

Atomic file writes. Files are written to a .tmp path and renamed on completion. If the process is interrupted mid-download the temp file is left on disk but never passed to the ingestion pipeline. A subsequent sync attempt will re-download the file cleanly.

State file as source of truth. Storing sync state as a JSON file on disk rather than in the database keeps the crawl subsystem self-contained. The state file can be inspected, backed up, or deleted independently of the application database. Deleting it forces a full re-ingest on the next run.

No automatic deletion of stale documents. Documents that disappear from a source's listing page are marked stale: true in state but are not removed from the pgvector index. Clinical guidelines are sometimes temporarily removed from listing pages due to review or update processes and may reappear. Silent automatic deletion of indexed clinical content is considered a higher risk than retaining a stale document until an admin explicitly removes it.

Disabled by default. The feature requires GUIDELINE_SYNC_ENABLED=true to activate. This prevents unintended crawl traffic in development, CI, and staging environments where the environment variable is not set.
