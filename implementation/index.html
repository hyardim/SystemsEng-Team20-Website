Implementation

How the application is built in code

The implementation spans three major codebases inside the project: the React frontend, the FastAPI backend, and the FastAPI RAG service. Docker Compose ties those pieces to a shared PostgreSQL/pgvector runtime. This page summarises how those modules map onto the final behaviour visible in the system.

Frontend implementation

The frontend router defines public pages, GP pages, specialist pages, admin pages, and a shared profile page. Auth state is persisted in localStorage and restored into React context so page reloads do not immediately lose route protection state.

Routing model

Protected route wrappers restrict views to allowed role sets such as GP/admin or specialist/admin.

Operational pages

GP consultation views, specialist review pages, and admin dashboards are implemented as separate page modules.

Notifications

A dropdown component polls every 15 seconds, tracks unread counts, and deep-links users into related consultations.

Admin charts

Recharts components visualise consultation states, specialty distribution, and daily AI query volumes.

Backend implementation

Application bootstrap

The backend creates ORM metadata at startup, ensures selected schema columns exist, and optionally seeds demo users.

Routers

The main app includes auth, chats, specialist, notifications, admin, and RAG-related routers.

Persistence

SQLAlchemy models represent users, chats, messages, notifications, audit logs, and file attachments.

Security

JWT authentication and password hashing are part of the core backend service layer.

Workflow logic lives primarily in service modules. GP messaging persists messages and triggers background AI generation. Specialist services manage assignment, review, revision, manual replacement, audit logging, and notification generation. Admin services aggregate statistics and expose oversight endpoints.

RAG service implementation

Startup path

The service loads the embedding model once, infers vector dimensionality, initialises pgvector schema, and warms the local model.

Answer path

/answer embeds the query, retrieves chunks, filters them, builds a grounded prompt, and returns answer plus citation sets.

Revision path

/revise uses the original query plus specialist feedback to regenerate a grounded answer.

Deployment implementation

Docker Compose defines the four primary runtime containers and their network. The frontend exposes port 3000, the backend exposes 8000, the RAG service exposes 8001, and PostgreSQL exposes 5432. Volume mounts are used for hot-reload paths and for making RAG data/configuration available inside containers.

Service Main responsibility Key runtime details
db_vector Application and vector database pgvector/pgvector:pg16 image, health checks, shared volume.
backend App API and workflow control Depends on database health, receives DATABASE_URL and RAG service URL.
rag_service Retrieval and generation engine Mounts data/configs, depends on database, reaches host model via host.docker.internal.
frontend SPA user interface Built via frontend Dockerfile, mapped to host port 3000.

Known implementation limitations

  • An older frontend API client remains in the repository even though a newer service path is in use.
  • A backend health router exists outside the main router list and appears unused in the primary runtime.
  • File attachments are modelled in the data layer but not fully integrated into the dominant consultation workflow.
  • Dual RAG API paths exist, reflecting an evolving design between deployed behaviour and modularised test targets.
  • Schema evolution currently uses startup helpers in addition to ORM table creation rather than a strict migration-only discipline.

Implementation emphasis

The codebase’s strongest feature is the workflow integration between retrieval-backed AI generation and specialist-controlled review, rather than raw model experimentation.
