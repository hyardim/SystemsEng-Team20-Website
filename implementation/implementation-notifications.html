Overview

Notification behavior is centralized in a repository layer that handles create/list/read transitions, while the frontend periodically refreshes unread counts and renders a consistent badge and dropdown experience. The backend caches both lists and unread counts with explicit invalidation so frequent polling does not overload the database. This keeps query logic isolated and UI behavior predictable even under heavy activity.

In a multi-role workflow, missed events can break collaboration. Reliable notifications ensure GPs, specialists, and admins see action-required updates quickly, reducing delays in review and response cycles.

Each NotificationType value corresponds to a specific workflow event. CHAT_ASSIGNED is triggered when a specialist claims a submitted consultation; the GP receives this notification and sees it as confirmation that their query is being reviewed. SPECIALIST_MSG is triggered when a specialist sends a comment on a consultation without finalizing the review; the GP sees a notification prompting them to check the chat for interim feedback or a request for clarification. CHAT_APPROVED is triggered when a specialist approves the AI response or provides a manual response closing the consultation; the GP sees this as confirmation that a reviewed answer is ready to act on. CHAT_REJECTED is triggered when a specialist rejects a consultation, typically because it was submitted in the wrong specialty or lacks sufficient clinical context; the GP is notified to revise and resubmit. CHAT_REVISION is triggered when a specialist requests AI regeneration on a consultation; the GP sees this as a status update that the AI response is being revised based on specialist feedback and a new version will appear shortly.

Repository Pattern

All database operations for notifications go through a repository module, keeping query logic out of the service layer:

# backend/src/repositories/notification_repository.py

def create(db, *, user_id, type, title, body=None, chat_id=None) -> Notification:
    notif = Notification(user_id=user_id, type=type, title=title,
                         body=body, chat_id=chat_id)
    db.add(notif)
    db.commit()
    db.refresh(notif)
    return notif

def list_for_user(db, user_id, *, unread_only=False) -> list[Notification]:
    query = db.query(Notification).filter(Notification.user_id == user_id)
    if unread_only:
        query = query.filter(Notification.is_read.is_(False))
    return query.order_by(Notification.created_at.desc()).all()

def mark_all_read(db, user_id) -> int:
    count = (db.query(Notification)
               .filter(Notification.user_id == user_id,
                       Notification.is_read.is_(False))
               .update({"is_read": True}))
    db.commit()
    return count

def count_unread(db, user_id) -> int:
    return (db.query(Notification)
              .filter(Notification.user_id == user_id,
                      Notification.is_read.is_(False))
              .count())

Keeping this logic in a repository avoids duplicating filters and sort semantics across multiple services. It also makes notification behavior easier to test in isolation from higher-level workflow actions.

Frontend: Notification Badge

The navigation bar polls the unread count every 30 seconds and shows a badge:

// frontend/src/hooks/useNotifications.ts (simplified)
export function useNotifications() {
  const [unreadCount, setUnreadCount] = useState(0);

  useEffect(() => {
    const fetchCount = async () => {
      const count = await getUnreadNotificationCount();
      setUnreadCount(count);
    };
    fetchCount();
    const interval = setInterval(fetchCount, 30_000);
    return () => clearInterval(interval);
  }, []);

  return { unreadCount };
}

The hook does an immediate fetch on mount before starting interval polling, so the badge is current as soon as navigation renders instead of waiting up to 30 seconds for first update.

// In NavigationBar.tsx
{unreadCount > 0 && (
  <span className="absolute -top-1 -right-1 bg-red-500 text-white
                   text-xs rounded-full w-4 h-4 flex items-center justify-center">
    {unreadCount > 9 ? "9+" : unreadCount}
  </span>
)}

The 9+ cap protects layout stability in tight nav spaces while still conveying “more than a few pending items.”

Notification Caching Pattern

Notifications are read-heavy and write-infrequent, so service methods use cache-aside reads with separate keys for list payloads and unread-count values. This is especially important because unread counts are polled every 30 seconds in the navigation bar.

# backend/src/services/notification_service.py
def list_notifications(db, user, *, unread_only=False) -> list[NotificationResponse]:
    cache_key = cache_keys.notifications(user.id, unread_only=unread_only)
    cached = cache.get_sync(cache_key, user_id=user.id, resource="notifications")
    if cached is not None:
        return [NotificationResponse(**item) for item in cached]

    notifs = notification_repository.list_for_user(db, user.id, unread_only=unread_only)
    response = [_to_response(n) for n in notifs]
    cache.set_sync(
        cache_key,
        [item.model_dump() for item in response],
        ttl=settings.CACHE_NOTIFICATION_TTL,
        user_id=user.id,
        resource="notifications",
    )
    return response

Unread count and notification list are cached independently so the frequent badge poll does not repeatedly execute COUNT(*) queries while still allowing full list fetches when dropdowns open.

A shared invalidation helper runs after every producer/consumer action: specialist review outcomes, specialist comments, mark-as-read, and mark-all-read. It clears unread-only and full-list variants together, keeping badge and dropdown views consistent immediately after writes.

Polling was chosen over a persistent WebSocket connection for notifications because notification events are low-frequency and latency-tolerant. In normal clinical workflows, a specialist may review a consultation and the GP receiving that notification can act on it within a minute or two — the precise delivery time within a 30-second window is not clinically significant. A WebSocket connection requires the server to maintain a persistent TCP connection for every logged-in user, which adds memory and file descriptor overhead that scales linearly with active users. For a relatively low-traffic platform where users are GPs checking results periodically throughout their day rather than trading real-time messages, the cost of maintaining per-user persistent connections outweighs the latency benefit. A 30-second polling interval delivers adequate responsiveness without that infrastructure overhead, and the cached unread-count response keeps individual poll requests cheap to serve.
