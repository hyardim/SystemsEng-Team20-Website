Overview

The frontend is organized around role-aware routing, a centralized auth context, a shared API client, and reusable feature components. This keeps data flow consistent from login through consultation creation, streaming, specialist review, and admin operations. SSE hooks and secure storage keep real-time updates and session state consistent across reloads.

That structure matters because product complexity grows quickly in multi-role clinical workflows. A clear architecture lowers regression risk, speeds onboarding for new contributors, and makes behavior easier to reason about when debugging production issues.

Application Routing

App.tsx defines 22 routes split across four role-based areas. All non-public routes are wrapped in ProtectedRoute which reads the current user's role from AuthContext and redirects to /access-denied if the role does not match:

All page components are lazy-loaded via React.lazy() so only the code for the current route is downloaded on first navigation.

Secure Storage

secureStorage.ts wraps localStorage with AES encryption so that user data and tokens stored client-side are not readable in plaintext via browser devtools or XSS. The encryption key is derived from a build-time env var (VITE_STORAGE_KEY). All AuthContext reads and writes go through this wrapper.

Auth State (AuthContext)

Global authentication state is managed via React Context. AuthContext holds the current user object and exposes login / logout helpers. State is persisted to secureStorage so the user remains logged in across page refreshes:

// frontend/src/contexts/AuthContext.tsx (simplified)
interface AuthContextValue {
  user: User | null;
  login: (credentials: LoginPayload) => Promise<void>;
  logout: () => void;
}

export const AuthProvider = ({ children }) => {
  const [user, setUser] = useState<User | null>(
    secureStorage.get("user")   // rehydrate from encrypted localStorage
  );

  const login = async (credentials) => {
    const { user } = await api.post("/auth/login", credentials);
    secureStorage.set("user", user);
    setUser(user);
  };

  const logout = () => {
    secureStorage.remove("user");
    setUser(null);
    navigate("/login");
  };

  return <AuthContext.Provider value={{ user, login, logout }}>{children}</AuthContext.Provider>;
};

The useAuth() hook is the single point of access to auth state across all components — no prop drilling.

API Client & Token Refresh Singleton

The frontend uses one shared API module for auth header injection, 401 refresh handling, and response/error normalization. Token headers are computed at call time so newly issued access tokens are always used on retry.

A module-level refreshInFlight promise coordinates concurrent 401 failures. When one request triggers refresh, all other failing requests await the same promise rather than issuing competing refresh calls.

// frontend/src/services/api.ts
let refreshInFlight: Promise<boolean> | null = null;

async function refreshSessionRequest(): Promise<boolean> {
  if (!refreshInFlight) {
    refreshInFlight = (async () => {
      const res = await fetch(apiUrl('/auth/refresh'), {
        method: 'POST',
        credentials: 'include',
      });
      if (res.ok) persistSession(await res.json());
      return res.ok;
    })().finally(() => { refreshInFlight = null; });
  }
  return refreshInFlight;
}

The key detail in this snippet is the finally reset. Without resetting refreshInFlight after resolve/reject, the client could become permanently stuck awaiting an old promise and never attempt refresh again.

After refresh completion, the singleton is reset so the next expiry cycle can start cleanly. Login and registration calls explicitly skip auth-refresh retries because 401 there means invalid credentials, not expired sessions.

The response handler also normalizes FastAPI 422 validation payloads (array-form detail) and plain string errors into display-ready messages, preventing raw object output in forms.

useChatStream Hook

useChatStream encapsulates the full SSE lifecycle for a single chat. Components mount it to start receiving tokens and unmount to clean up:

// frontend/src/hooks/useChatStream.ts (simplified)
export function useChatStream(chatId: number) {
  const [streamingContent, setStreamingContent] = useState("");
  const [isStreaming, setIsStreaming] = useState(false);

  useEffect(() => {
    const cleanup = subscribeToChatStream(chatId, {
      onStreamStart: () => setIsStreaming(true),
      onContent:     (content) => setStreamingContent(content),  // cumulative
      onComplete:    () => setIsStreaming(false),
      onError:       () => setIsStreaming(false),
    });
    return cleanup;   // closes EventSource on unmount
  }, [chatId]);

  return { streamingContent, isStreaming };
}

Using cumulative content updates keeps rendering idempotent: each incoming event represents full text-so-far, so dropped intermediate events do not corrupt local message state.

GPQueryDetailPage and SpecialistQueryDetailPage both consume this hook. The onContent callback receives cumulative content, so message text can be replaced directly without client-side token concatenation drift.

The production implementation runs a five-phase state machine: idle → connecting → streaming → completed with fallback_polling as the error/timeout branch. Polling backoff starts from a short interval and grows to a capped maximum to avoid aggressive request loops during degraded SSE conditions.

Connection cleanup is centralized through a cleanup ref so reconnections do not create duplicate subscriptions. Route changes (chatId dependency changes) trigger explicit teardown and reset to idle, preventing stream leakage between consultations.

The hook accepts a setMessages updater from the page rather than owning message state itself. This keeps page-level data ownership explicit and avoids stale-closure update bugs by always using functional state updates.

Frontend Input Validation & Error Messages

Forms apply fast client-side validation before requests, while backend Pydantic validation remains the source-of-truth fallback. Field errors are collected per input so users see all issues in one pass rather than one-at-a-time failure loops.

Validation areaRuleUser feedback
EmailTrim/normalize and regex format check.Inline error under email field.
PasswordMin length + upper/lower/digit/special requirements.Inline error plus strength meter guidance.
Confirm passwordMust match password exactly.Specific mismatch message.
Specialist profile fieldsConditionally required for specialist role.Role-aware inline validation.

Validation errors are cleared per field on change so stale messages disappear as the user fixes input. This avoids the common UX issue where corrected fields still display outdated errors.

Inline fieldErrors and top-level error banners serve different failure classes: field-specific issues appear under controls, while non-field API failures are surfaced in a global alert region.

FastAPI 422 payloads are normalized into readable strings to avoid surfacing raw object structures in the UI.

Form, Data, and Error State Management

Pages follow a consistent three-tier state model: loading for initial page fetch lifecycle, data state for rendered resources, and actionLoading for mutations. This separation prevents double-submits and keeps refresh behavior predictable under slow network conditions.

Data state uses explicit nullable defaults and guarded rendering paths, so missing entities render deterministic fallbacks rather than undefined-access failures. After most mutations, pages reconcile via refetch rather than broad optimistic patching to avoid drift from server truth.

Error state is cleared at action start and set from normalized API responses on failure, so stale errors do not persist across attempts. Abort-controller cleanup prevents setState-on-unmounted warnings when route changes interrupt in-flight requests.
