Overview

The backend maps domain entities (User, Chat, Message, etc.) into typed SQLAlchemy models and then leverages PostgreSQL features like ENUM, JSONB, and targeted indexes to enforce structure and speed up common queries. A sync session serves normal API requests, while a separate async session supports AI generation workloads without blocking the event loop. Alembic tracks schema changes so migrations are repeatable and safe across environments.

This layout keeps data correct under heavy concurrency, makes GP and specialist workflows reliable, and avoids ad-hoc database edits that can drift between deployments. It also ensures performance-critical queries have matching indexes so response times stay stable as data volume grows.

Dual-Mode Session Configuration

The application uses two database sessions — synchronous for regular API routes, and asynchronous for the AI generation background task (which must not block the event loop):

# backend/src/db/session.py
from sqlalchemy import create_engine
from sqlalchemy.ext.asyncio import create_async_engine, async_sessionmaker

# ── Synchronous engine — all standard API endpoints ──────────────────────────
engine = create_engine(
    DATABASE_URL,
    pool_size=20,        # keep 20 connections warm
    max_overflow=30,     # allow up to 30 extra connections under load
    pool_timeout=30,
    pool_recycle=1800,   # recycle connections every 30 min to avoid stale TCP
    pool_pre_ping=True,  # check liveness before handing connection to a request
)
SessionLocal = sessionmaker(bind=engine, autocommit=False, autoflush=False)

def get_db() -> Iterator[Session]:
    db = SessionLocal()
    try:
        yield db          # FastAPI injects this via Depends(get_db)
    except Exception:
        db.rollback()     # auto-rollback on unhandled exception
        raise
    finally:
        db.close()

# ── Async engine — AI generation background task ──────────────────────────────
ASYNC_DATABASE_URL = DATABASE_URL.replace("postgresql://", "postgresql+asyncpg://")

async_engine = create_async_engine(ASYNC_DATABASE_URL, pool_size=20, max_overflow=30,
                                    pool_pre_ping=True)
AsyncSessionLocal = async_sessionmaker(bind=async_engine, expire_on_commit=False)

async def get_async_db() -> AsyncIterator[AsyncSession]:
    async with AsyncSessionLocal() as session:
        try:
            yield session
        except Exception:
            await session.rollback()
            raise

The sync and async engines intentionally use similar pool settings so behavior under load is predictable across API and generation paths. pool_pre_ping and pool_recycle reduce stale-connection errors after idle periods or network interruptions.

Rollback-on-exception in both dependencies ensures failed requests do not leave partially applied writes in transaction scope. This is especially important when service methods compose multiple repository operations.

expire_on_commit=False on async sessions prevents objects used immediately after commit from triggering lazy reloads during event-loop-sensitive generation flows.

Declarative Base

All ORM models inherit from one shared SQLAlchemy Base class. This gives Alembic a single metadata graph to inspect when auto-generating migrations.

# backend/src/db/base.py
from sqlalchemy.orm import DeclarativeBase

class Base(DeclarativeBase):
    pass

Using one declarative base also keeps table naming, relationship metadata, and migration autogeneration coherent. Splitting models across different base classes is a common source of missing-table migration bugs.

User Model

The User model is the identity anchor for the whole platform. It stores authentication state, role-based access control attributes, and profile metadata used by routing and permission checks across GP, specialist, and admin workflows.

Indexing email, role, and specialty keeps common lookups fast (login, role filtering, specialist matching), while fields like session_version and email_verified support secure token invalidation and account verification policies.

# backend/src/db/models/user.py
class UserRole(str, enum.Enum):
    GP         = "gp"
    SPECIALIST = "specialist"
    ADMIN      = "admin"

class User(Base):
    __tablename__ = "users"
    __table_args__ = (
        Index("ix_users_email",    "email"),
        Index("ix_users_role",     "role"),
        Index("ix_users_specialty","specialty"),
    )

    id:                  Mapped[int]             = mapped_column(Integer, primary_key=True)
    email:               Mapped[str]             = mapped_column(String, unique=True, nullable=False)
    hashed_password:     Mapped[str]             = mapped_column(String, nullable=False)
    full_name:           Mapped[str | None]      = mapped_column(String, nullable=True)
    role:                Mapped[UserRole]        = mapped_column(SQLEnum(UserRole), default=UserRole.GP)
    specialty:           Mapped[str | None]      = mapped_column(String, nullable=True)
    is_active:           Mapped[bool]            = mapped_column(Boolean, default=True)
    email_verified:      Mapped[bool]            = mapped_column(Boolean, default=False)
    session_version:     Mapped[int]             = mapped_column(Integer, default=0)
    created_at:          Mapped[datetime]        = mapped_column(DateTime, default=utc_now)

    chats:          Mapped[list["Chat"]]         = relationship("Chat", foreign_keys="Chat.user_id")
    assigned_chats: Mapped[list["Chat"]]         = relationship("Chat", foreign_keys="Chat.specialist_id")
    notifications:  Mapped[list["Notification"]] = relationship("Notification", back_populates="user")

Chat Model

The Chat model drives the entire consultation lifecycle. status is a PostgreSQL ENUM preventing invalid values at the database level. patient_context uses JSONB for flexible metadata without extra columns or migrations. Multiple composite indexes support the most common query patterns.

# backend/src/db/models/chat.py
class ChatStatus(str, enum.Enum):
    OPEN      = "open"
    SUBMITTED = "submitted"
    ASSIGNED  = "assigned"
    REVIEWING = "reviewing"
    APPROVED  = "approved"
    REJECTED  = "rejected"
    CLOSED    = "closed"
    FLAGGED   = "flagged"
    ARCHIVED  = "archived"

class Chat(Base):
    __tablename__ = "chats"
    __table_args__ = (
        Index("ix_chats_user_id",               "user_id"),
        Index("ix_chats_specialist_id",          "specialist_id"),
        Index("ix_chats_status",                 "status"),
        Index("ix_chats_user_archived_created_at","user_id","is_archived","created_at"),
        Index("ix_chats_status_specialty_created_at","status","specialty","created_at"),
        Index("ix_chats_specialist_status_assigned_at",
              "specialist_id","status","assigned_at"),
    )

    id:              Mapped[int]             = mapped_column(Integer, primary_key=True)
    title:           Mapped[str]             = mapped_column(String, default="New Chat")
    status:          Mapped[ChatStatus]      = mapped_column(SQLEnum(ChatStatus), default=ChatStatus.OPEN)
    specialty:       Mapped[str | None]      = mapped_column(String, nullable=True)
    patient_context: Mapped[dict | None]     = mapped_column(JSONB, nullable=True)
    specialist_id:   Mapped[int | None]      = mapped_column(ForeignKey("users.id", ondelete="SET NULL"))
    assigned_at:     Mapped[datetime | None] = mapped_column(DateTime, nullable=True)
    review_feedback: Mapped[str | None]      = mapped_column(Text, nullable=True)
    is_archived:     Mapped[bool]            = mapped_column(Boolean, default=False)
    user_id:         Mapped[int]             = mapped_column(ForeignKey("users.id", ondelete="CASCADE"))

    owner:     Mapped["User | None"]         = relationship("User", foreign_keys=[user_id])
    specialist: Mapped["User | None"]        = relationship("User", foreign_keys=[specialist_id])
    messages:  Mapped[list["Message"]]       = relationship("Message", cascade="all, delete-orphan")
    files:     Mapped[list["FileAttachment"]] = relationship("FileAttachment", cascade="all, delete-orphan")

Each ChatStatus value represents a distinct phase in the consultation lifecycle with defined triggers and permissions. OPEN is the initial state when a chat is created; it transitions to SUBMITTED automatically when the GP sends the first message (auto-submit path) or explicitly via the submit action. SUBMITTED means the consultation is in the specialist queue awaiting assignment; any available specialist with the matching specialty can claim it. ASSIGNED is set when a specialist claims the consultation, recording both the specialist's identity and the assignment timestamp; only the assigned specialist can act on it from this point. REVIEWING is entered when the specialist begins per-message review actions such as editing or requesting changes, allowing multiple review passes without finalizing the consultation outcome. APPROVED is set by the specialist when they accept the AI response or provide a manual response; this is the terminal positive state and triggers a notification to the GP. REJECTED is set when the specialist determines the consultation cannot proceed as submitted, sending the chat back with feedback; the GP receives a notification and can revise and resubmit. CLOSED is set by the GP when they close a consultation before specialist review, typically because the clinical question was resolved through other means. FLAGGED is reserved for admin moderation, allowing abnormal consultations to be marked for review without altering the normal status flow. ARCHIVED is a soft-delete state set when a GP archives a consultation; archived chats are excluded from default list queries but retained in the database for audit purposes and can be queried explicitly with the is_archived filter.

The ondelete and relationship choices are intentional safety policies: deleting a GP cascades owned chats, deleting a specialist does not delete consultations (SET NULL), and child message/file rows are controlled through chat ownership lifecycle.

This combination avoids accidental data loss in specialist-account changes while keeping orphaned child rows from accumulating when consultations are removed.

Message Model

citations is stored as JSONB — each element contains document title, source name, section path, and page numbers, avoiding a separate citations table. is_error messages are excluded from conversation history so failed generation attempts don't pollute the LLM's context.

# backend/src/db/models/message.py
class Message(Base):
    __tablename__ = "messages"
    __table_args__ = (
        Index("ix_messages_chat_id",                       "chat_id"),
        Index("ix_messages_chat_created_at",               "chat_id","created_at"),
        Index("ix_messages_chat_sender_review_created_at", "chat_id","sender","review_status","created_at"),
    )

    id:              Mapped[int]          = mapped_column(Integer, primary_key=True)
    content:         Mapped[str | None]   = mapped_column(Text)
    sender:          Mapped[str]          = mapped_column(String)   # "user" | "ai" | "specialist"
    created_at:      Mapped[datetime]     = mapped_column(DateTime, default=utc_now)
    citations:       Mapped[list | None]  = mapped_column(JSONB(none_as_null=True), nullable=True)
    is_generating:   Mapped[bool]         = mapped_column(Boolean, default=False)
    is_error:        Mapped[bool]         = mapped_column(Boolean, default=False)
    review_status:   Mapped[str | None]   = mapped_column(String, nullable=True)
    review_feedback: Mapped[str | None]   = mapped_column(Text, nullable=True)
    chat_id:         Mapped[int]          = mapped_column(ForeignKey("chats.id"), nullable=False)

Message flags (is_generating, is_error, review_status) are not cosmetic metadata; they drive specialist action guards, streaming placeholders, and conversation-history inclusion rules.

The composite chat/sender/review index supports “latest unreviewed AI message” access patterns without scanning entire message timelines.

Notification Model

The notification table stores per-user workflow events with unread/read state and optional chat linkage. Composite indexes optimize badge count polling and dropdown list rendering.

# backend/src/db/models/notification.py
class NotificationType(str, enum.Enum):
    CHAT_ASSIGNED  = "chat_assigned"
    SPECIALIST_MSG = "specialist_msg"
    CHAT_APPROVED  = "chat_approved"
    CHAT_REJECTED  = "chat_rejected"
    CHAT_REVISION  = "chat_revision"

class Notification(Base):
    __tablename__ = "notifications"
    __table_args__ = (
        Index("ix_notifications_user_id", "user_id"),
        Index("ix_notifications_is_read", "is_read"),
        Index("ix_notifications_user_created_at", "user_id", "created_at"),
        Index("ix_notifications_user_read_created_at", "user_id", "is_read", "created_at"),
    )

    id: Mapped[int] = mapped_column(Integer, primary_key=True)
    user_id: Mapped[int] = mapped_column(ForeignKey("users.id", ondelete="CASCADE"))
    type: Mapped[NotificationType] = mapped_column(SQLEnum(NotificationType))
    title: Mapped[str] = mapped_column(String, nullable=False)
    is_read: Mapped[bool] = mapped_column(Boolean, default=False)

Database Migrations (Alembic)

Schema changes are applied via Alembic migration scripts. Some require raw SQL that cannot be auto-detected — for example, adding the full-text search vector column:

# Auto-detect model changes and create a migration
alembic revision --autogenerate -m "add is_archived to chats"

# Apply all pending migrations
alembic upgrade head
-- migrations/003_add_text_search_vector.sql
ALTER TABLE rag_chunks
    ADD COLUMN IF NOT EXISTS text_search_vector tsvector
    GENERATED ALWAYS AS (to_tsvector('english', text)) STORED;

CREATE INDEX IF NOT EXISTS idx_rag_chunks_text_search
    ON rag_chunks USING GIN (text_search_vector);

GENERATED ALWAYS AS ... STORED is a PostgreSQL 12+ feature — the column is automatically recomputed by the database engine whenever text changes with no application code required.

Legacy Database Detection

bootstrap.py inspects the existing schema and detects unstamped databases — where backend tables already exist but no alembic_version row is present. Instead of crashing mid-migration it fails early with a clear recovery message:

# backend/src/db/bootstrap.py
MANAGED_BACKEND_TABLES = frozenset({
    "audit_logs", "chats", "email_verification_tokens",
    "file_attachments", "messages", "notifications",
    "password_reset_tokens", "users",
})

def _ensure_supported_migration_state() -> None:
    existing = _existing_public_tables()
    managed_present = existing & MANAGED_BACKEND_TABLES
    if managed_present and "alembic_version" not in existing:
        raise RuntimeError(
            "Legacy unstamped database detected. "
            "Run: alembic stamp head  then  alembic upgrade head"
        )

Fail-fast legacy detection prevents a dangerous partial-migration state where some DDL runs before Alembic discovers version mismatches. Operators get an explicit recovery command sequence instead of ambiguous migration stack traces.

SQL Indexes, Unique Constraints & Database Normalisation

Critical uniqueness is enforced at the database layer, not just in API validation. For example, users.email is unique, and rag_chunks uses a composite uniqueness rule on (doc_id, doc_version, chunk_id) to guarantee deterministic upserts.

The schema follows 3NF: users, chats, messages, notifications, and file attachments are separate tables linked by foreign keys, while consultation-specific context remains in JSONB on chats.

Index rollout was staged across migrations: baseline identity/auth indexes, then workload-driven single-column indexes, then composite production-path indexes tuned for specialist queues, GP timeline reads, notification badge counts, and admin audit filtering.

Composite indexes with concurrent creation in PostgreSQL were used for larger deployments to reduce locking risk during migration.

Each composite index is matched to a specific production query shape. ix_chats_user_archived_created_at accelerates the GP's own consultation list query, which filters by user_id to scope to a single GP, optionally filters by is_archived to show only active consultations, and orders results by created_at descending — without the index, this query performs a full table scan ordered by timestamp on every page load. ix_chats_status_specialty_created_at accelerates specialist queue queries, which filter by status (typically SUBMITTED or ASSIGNED) and optionally by specialty to show only relevant domain work, ordered by submission time so oldest cases appear first; this is the highest-frequency query on the specialist-facing side of the platform. ix_messages_chat_sender_review_created_at supports "latest unreviewed AI message" lookups used by the specialist review guard, which must find the most recent AI message for a given chat that has not yet been reviewed — without the index this requires a full timeline scan of all messages for the chat. ix_notifications_user_read_created_at supports both the unread badge count poll (filtering by user_id and is_read = false) and the notification dropdown (filtering and ordering by created_at), which together are the most frequently executed notification queries given the 30-second polling interval. ix_audit_logs_action_timestamp makes admin audit filtering by action type and date range fast enough to use interactively without a separate reporting database.
