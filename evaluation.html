Evaluation

How well the system meets its stated requirements

This page evaluates Ambience-AI-1.5 against its original MoSCoW requirements, provides a critical assessment of each major technical dimension, identifies known limitations, and documents the team's contributions. All Must, Should, and Could requirements were fully met. Codebase verification confirmed implementations across all 19 in-scope requirements.

Critical evaluation

Each major technical dimension is rated on a 5-point scale with a brief justification. Ratings reflect engineering quality and delivery confidence, not production readiness for clinical deployment.

RAG grounding & citations
4 / 5 pgvector retrieval with citation structures is solid engineering. No formal clinical golden-set validation; accuracy claims remain prototype-level.
Specialist review workflow
5 / 5 End-to-end consultation lifecycle embedded in the state machine from the start, not bolted on. Approve / edit / reject / revise paths all implemented.
Role-based access control
4 / 5 JWT authentication with role-scoped routes and protected API paths. No MFA or session revocation; acceptable for prototype scope.
Test coverage & quality
5 / 5 Layered strategy: unit → integration → Playwright E2E. 90 %+ threshold enforced in CI. 96+ test files, 20 000+ lines of test code.
Frontend usability
4 / 5 Clean role-differentiated views, notification polling, Recharts dashboards. No formal WCAG audit; keyboard navigation not fully validated.
Containerised deployment
4 / 5 7-container Docker Compose stack with health checks, nginx TLS reverse proxy, and Alembic migrations. Documented startup and predictable service boundaries.
Audit logging & observability
4 / 5 Structured audit trail for every specialist decision; admin dashboard surfaces statistics. No external monitoring (Prometheus / alerting) in scope.
Failure handling & resilience
4 / 5 Graceful fallbacks on retrieval / model failure implemented and tested. Local model dependency (Ollama) remains the main single point of failure.

Known issues & limitations

The following limitations are known and acknowledged. They do not undermine the core prototype goals but would need to be addressed before any production deployment.

No formal clinical validation

Accuracy claims about RAG answers have not been evaluated by domain experts against a golden test set. The system is an engineering prototype, not a clinically validated tool.

No MFA or session revocation

Authentication uses JWT with a fixed expiry. There is no multi-factor authentication or token revocation list. Acceptable for a prototype but required for clinical deployment.

Local model dependency

The RAG service depends on a locally running Ollama instance. If the model server is unavailable, the fallback path is safe but the system cannot generate grounded answers.

Future work

The following directions represent the highest-value investments for a next iteration of Ambience-AI-1.5, ordered roughly by impact on production readiness.

🏥 Clinical expert evaluation

Engage NHS GPs and specialists in structured usability and accuracy evaluation. Build a golden-set of consultation queries with expert-validated reference answers to measure RAG answer quality formally.

🔐 MFA & session management

Add multi-factor authentication, refresh-token rotation, and a token revocation list. These are baseline requirements for any NHS-adjacent clinical deployment.

📊 External monitoring & alerting

Integrate Prometheus metrics and Grafana dashboards. Add structured logging export to allow SRE-style alerting on latency regressions, retrieval failures, and model unavailability.

Overall verdict

Assessment summary

Ambience-AI-1.5 is a technically credible full-stack prototype for guideline-grounded clinical consultation support. All Must, Should, and Could requirements were fully delivered — confirmed by direct codebase inspection. The specialist review workflow, real-time SSE streaming, and layered test strategy are the strongest engineering contributions. The primary gap is the absence of formal clinical validation, necessary before making stronger claims about real-world utility. The codebase is well-structured, modular, and deployable; the architecture supports the next iteration described in future work.
