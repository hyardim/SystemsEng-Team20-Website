Project background

The challenge

GPs often need quick answers to routine Neurology and Rheumatology questions, yet relevant NICE and BSR guidance is spread across long documents. We also had to respect clinical safety: unsupported answers or missing citations were unacceptable, and some early retrievals lacked the right section granularity.

Our approach

We built a retrieval-first workflow that prioritises guideline evidence, then generates structured responses with citations and an escalation path. We iterated chunking, retrieval thresholds, and response formatting based on review feedback to reduce ambiguity and improve confidence alignment.

Project goals

Technical goals
  • Deploy a domain-adapted Med42 runtime with efficient inference performance on local hardware.
  • Build a retrieval pipeline that surfaces the correct NICE and BSR guideline sections.
  • Implement safety mechanisms for refusal when evidence is insufficient.
  • Provide a clean, accessible web interface aligned with NHS usability guidance.
Non-technical goals
  • Improve GP access to accurate, guideline-based answers for routine queries.
  • Reduce repetitive specialist workload through safe automation and escalation.
  • Maintain trust through visible citations and clear evidence linkage.
  • Ensure outputs remain clinically appropriate and easy to review.

Requirement gathering

We gathered requirements through weekly stakeholder meetings, one semi-structured interview, and a short validation survey. This section directly answers how we collected requirements, whether we designed a survey, and how we analysed the data.

How did we collect requirements?

We combined three sources: (1) weekly meetings with supervisors/partners, (2) a semi-structured interview with a GP-style user profile, and (3) guideline-driven checks against NICE/BSR content.

Did we design a survey?

Yes. We designed a short survey with 7 items: five Likert-scale questions (trust, clarity, speed, citation usefulness, and escalation clarity) and two open-text questions for concerns and missing features.

How did we analyse the survey data?

We summarised Likert responses with averages and agreement rates, then coded open-text feedback into themes (citation visibility, answer brevity, and safe escalation wording) to update requirement priorities.

Interview transcript summary (open each question)

The interview was semi-structured and focused on real consultation workflow pain points. Each question below expands to show what we learned and which requirement it informed.

Q1. When do you need support most during consultation?

The need is highest when time is limited and guideline sections are lengthy. This led to a must-have requirement for concise outputs with immediate citation anchors.

Q2. What makes an AI answer trustworthy in clinical contexts?

Trust depended on visible evidence and source dates. This informed the requirement to show guideline source metadata and explicit citation placement.

Q3. What should happen when evidence is weak or conflicting?

Users preferred refusal + escalation over speculative guidance. We therefore prioritised low-confidence refusal logic with a clear specialist handoff path.

Q4. How much detail is useful in the first response?

Interview feedback preferred short answers first, with optional deeper detail. This translated into a concise response format and structured follow-up prompts.

Q5. Which UI elements reduce cognitive load in busy sessions?

Clear headings, compact answer blocks, and obvious escalation controls were prioritised. This informed our usability and accessibility non-functional requirements.

Scope boundaries and exclusions

  • The platform is specialised for neurology and rheumatology guideline assistance rather than general-purpose medical chat.
  • The current workflow is predominantly text-based; file attachments are modelled in the backend but not fully wired into the main chat path.
  • Polling is used for updates instead of websockets, which simplifies deployment but trades off push responsiveness.
  • The project includes cloud-hosting notes and alternative model-routing concepts, but not all of them belong to the main runtime path.

Constraint summary

A coursework project must show end-to-end engineering depth, so the system balances a real deployment shape with pragmatic shortcuts such as startup schema patching and an evolving RAG module layout.
