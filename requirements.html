Requirements

Functional and non-functional requirements

The project requirements were driven by a clinical consultation use case rather than a generic conversational AI goal. The resulting system has to support evidence-grounded answers, multi-role review, and traceable operational behaviour under a realistic full-stack deployment model.

Problem definition

Primary care clinicians often need rapid answers for specialist-adjacent questions, but generic LLM chat interfaces are not sufficient for high-stakes, evidence-backed decision support. Ambience-AI-1.5 therefore focuses on guideline retrieval, transparent citations, and a workflow where specialists can intervene directly before a case is considered resolved.

Requirement framing

The system must behave like a consultation support workflow with AI assistance, not like an unbounded medical chatbot.

Stakeholders and user groups

Stakeholder Primary need System response
GP Ask a clinical question quickly and receive a grounded answer with sources. Protected GP routes, consultation creation, timeline view, and citation display.
Specialist Review AI output, control workflow quality, and intervene when guidance is weak. Assignment queue, review endpoints, revision path, and manual response capability.
Admin Observe operational health, manage users, and inspect audit history. Admin dashboards, user management endpoints, chat oversight, and audit log browsing.
Project team Deploy and iterate on a multi-service architecture within coursework constraints. Docker Compose orchestration, modular services, and automated tests.

Core functional requirements

Authentication and access control

JWT-based login, registration, password reset, and route-level role protection for GP, specialist, and admin users.

Consultation lifecycle

Chats must progress through open, submitted, assigned, reviewing, approved, or rejected states.

RAG-backed answer generation

User questions must be forwarded to a retrieval service that returns grounded answers plus citation metadata.

Specialist control path

Specialists must be able to self-assign, review whole chats or individual messages, request revisions, or manually replace answers.

Notifications and audit logs

Users need status updates while admins need traceability for major workflow and security events.

Admin analytics

The system must expose aggregate statistics over consultations, roles, AI responses, and RAG-grounded outputs.

Non-functional requirements

Groundedness

Answers should cite retrieved guideline material and avoid unsupported free-form generation.

Auditability

Major actions need server-side logs so admin users can inspect the workflow retrospectively.

Role isolation

Users should only see and execute actions permitted by their operational role.

Deployability

The stack should be runnable with Docker Compose on ports 3000, 8000, 8001, and 5432.

Extensibility

The architecture should permit alternate model endpoints and evolving RAG retrieval modules.

Clinical restraint

The system should prefer no-answer behaviour over hallucinated certainty when evidence quality is low.

Scope boundaries and exclusions

  • The platform is specialised for neurology and rheumatology guideline assistance rather than general-purpose medical chat.
  • The current workflow is predominantly text-based; file attachments are modelled in the backend but not fully wired into the main chat path.
  • Polling is used for updates instead of websockets, which simplifies deployment but trades off push responsiveness.
  • The project includes cloud-hosting notes and alternative model-routing concepts, but not all of them belong to the main runtime path.

Constraint summary

A coursework project must show end-to-end engineering depth, so the system balances a real deployment shape with pragmatic shortcuts such as startup schema patching and an evolving RAG module layout.

Success criteria

  • Grounded AI responses with citations.
  • Role-aware consultation and review workflow.
  • Deployable multi-service stack.
  • Automated tests across frontend, backend, and RAG paths.

Ambience-AI-1.5

A technical systems report for a university software engineering project. The system under study is a role-aware clinical consultation platform that combines React, FastAPI, PostgreSQL with pgvector, and a grounded medical RAG service.

This website intentionally describes implementation details, deployment paths, safeguards, testing, and known limitations rather than product marketing claims.
