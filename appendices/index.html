User manual

The appendix should help a marker or project partner get into the system quickly. For Ambience-AI-1.5 that means separating the GP, specialist, and admin journeys rather than describing the whole application as if every user sees the same interface.

User-access pack

The user manual should begin with the live application URL if one exists, explain what each role can do, and then walk through the most important screens in order. Screenshots are especially useful for the first-login flow, consultation creation, specialist review, and admin oversight views.

  • Show the reader where to log in and what credentials format to use.
  • Explain the happy path for each role instead of listing unrelated features.
  • Use screenshots with short captions to anchor the explanation visually.

Role 1

GP workflow

The GP journey should show how a consultation starts and where grounded answers appear.

  • Log in as a GP user and create a consultation with title, specialty, and severity.
  • Submit the first clinical question to trigger grounded AI generation.
  • Inspect the response, citations, and consultation status in the detailed timeline.
  • Track when the case moves into specialist review.
Role 2

Specialist workflow

The specialist path should focus on review decisions, evidence checking, and revisions.

  • Open the review queue and self-assign a relevant consultation.
  • Inspect AI responses and supporting citations in the review interface.
  • Approve, reject, request changes, or replace an AI answer manually.
  • Return a review outcome that the GP can follow.
Role 3

Admin workflow

The admin path should explain oversight rather than ordinary clinical usage.

  • View platform statistics for consultations, users, and AI activity.
  • Inspect user records and role distribution across the system.
  • Review chats globally when investigating operational issues.
  • Open audit logs to trace sensitive or security-relevant actions.
Screenshots to include

Login, consultation creation, specialist review, and admin oversight should all appear as labelled screenshots.

Live access details

If the system is deployed online, add the URL, browser assumptions, and any access restrictions explicitly.

Role-specific accounts

Provide one test account per role so the reader can verify each workflow without guessing permissions.

Deployment manual

Very important

This is the section that most directly affects handover quality. A project partner should be able to deploy the system from source without reverse-engineering the rest of the report.

Suggested deployment flow

Prepare prerequisites Confirm Docker is available, the database runtime is reachable, and the local or cloud LLM endpoint expected by the RAG service is configured.
Populate environment files Document the required environment values for frontend, backend, database, and RAG services before the reader starts any containers.
Launch the stack Start the frontend, backend, RAG service, PostgreSQL with pgvector, and any supporting services through Docker Compose.
Prepare data and migrations Explain how schema setup, seeded accounts, and guideline ingestion are handled before first use.
Verify health Check service status, confirm health endpoints respond, and log in using the supplied demo accounts.
Record difficult steps If anything is platform-specific or brittle, include screenshots so the handover is reproducible instead of implicit.

Core commands to document

Use explicit startup commands in the appendix instead of assuming prior knowledge.

git clone <application-repository> cd <project-root> cp <env-template> <env-file> docker compose up --build docker compose ps docker compose logs -f <service>

What the project partner should receive

  • A repository URL or archive for the main application codebase.
  • Environment templates and a safe handover path for secrets.
  • Instructions for seeded accounts, migrations, and document ingestion.
  • Notes on whether inference runs locally, in the cloud, or through a hybrid route.

Deployment evidence

If any setup step is awkward, add screenshots of the relevant terminal output, environment file shape, or first successful login. The goal is reproducibility, not just a conceptual outline.

Legal issues and processes

This section should not be a generic privacy statement. It should explain which legal notes apply, what data risks exist in this prototype, and where the current project still stops short of production-grade governance.

Use the provided legal notes

The notes already uploaded to Moodle should be used as the baseline structure for this appendix.

  • Reference the relevant course guidance instead of leaving the section vague.
  • Relate that guidance to this project's roles, audit trail, and access model.
  • State clearly which assumptions are academic and which controls are actually implemented.

Project-specific issues to cover

  • Consultation payloads and uploaded files may contain sensitive patient context.
  • External or cloud-hosted inference routes create data transfer and processor questions.
  • Audit logs, retention, and role-based access all affect the legal interpretation of the system.
  • Source licensing and generated clinical text should be framed with clear usage boundaries.

Boundary to state explicitly

Ambience-AI-1.5 should be presented as an academic prototype. The current design demonstrates privacy-aware architecture and role separation, but it is not a full GDPR compliance or clinical governance pack.

Common issues on the appendices page

Most appendix problems are not about theory. They come from missing access details, weak evidence, and a handover process that assumes too much of the reader.

01

Missing deployed URL

03

User manual lacks screenshots

Screenshots should support the manual for clarity, especially around login, consultation flow, review, and admin views.

Ambience-AI-1.5

This website intentionally describes implementation details, deployment paths, safeguards, testing, and known limitations rather than product marketing claims.
