Role 1

GP workflow

The GP journey should show how a consultation starts and where grounded answers appear.

  • Log in as a GP user and create a consultation with title, specialty, and severity.
  • Submit the first clinical question to trigger grounded AI generation.
  • Inspect the response, citations, and consultation status in the detailed timeline.
  • Track when the case moves into specialist review.
Role 2

Specialist workflow

The specialist path should focus on review decisions, evidence checking, and revisions.

  • Open the review queue and self-assign a relevant consultation.
  • Inspect AI responses and supporting citations in the review interface.
  • Approve, reject, request changes, or replace an AI answer manually.
  • Return a review outcome that the GP can follow.
Role 3

Admin workflow

The admin path should explain oversight rather than ordinary clinical usage.

  • View platform statistics for consultations, users, and AI activity.
  • Inspect user records and role distribution across the system.
  • Review chats globally when investigating operational issues.
  • Open audit logs to trace sensitive or security-relevant actions.

README

The README area should act as the document a developer reads first. It needs to state what Ambience-AI-1.5 is, how the services fit together, and where somebody should begin when running or extending the project.

Project overview

Ambience-AI-1.5 is a role-aware clinical consultation platform. The system joins a React frontend, a FastAPI backend, a separate RAG service, PostgreSQL with pgvector, Redis-backed background work, and containerised runtime tooling.

  • The frontend serves GP, specialist, and admin interfaces.
  • The backend handles auth, workflow state, notifications, and auditability.
  • The RAG service handles retrieval, reranking, and grounded answer generation.

GDPR

This section explains how Ambience-AI-1.5 should be understood from a GDPR perspective. The project is an academic prototype, but the workflow still raises real data-protection questions because clinical prompts, uploaded files, and review notes may contain sensitive health information if used carelessly.

Data Usage Statement

Ambience-AI-1.5 processes account information, consultation records, uploaded guideline material, notifications, and audit logs in order to support the GP, specialist, and admin workflows described elsewhere in this report. In a real deployment, consultation content could amount to personal data and, in many cases, special category health data. That means data entered into the system should be treated with a higher level of care than ordinary application content.

For the purposes of this university project, the system should be presented as a prototype and demonstrations should avoid unnecessary real patient-identifying information. The academic setting does not remove GDPR concerns; it only narrows the claim being made about deployment readiness.

Key GDPR Considerations

Anyone using or evaluating the prototype should keep the following principles in mind when entering or handling data:

  • Data minimisation: only the minimum necessary clinical context should be entered into prompts, notes, or uploaded supporting material.
  • Purpose limitation: consultation data should only be used to support the defined referral and review workflow, not unrelated experimentation.
  • Access control: GP, specialist, and admin accounts should only access the information required for their role.
  • External processing risk: if a prompt or document is routed to a cloud-hosted inference provider, that creates additional processor and transfer questions that would need formal review before real deployment.
  • Auditability and retention: logs and workflow records improve traceability, but they also require clear retention and governance rules in any production setting.

This means the current project should not be described as GDPR-complete. It shows awareness of the right concerns, but it does not replace a full compliance pack, organisational policy set, or legal review.

User Responsibility

By using the system in demonstrations, testing, or handover review, users should understand that they are responsible for avoiding unnecessary personal data and for respecting the access boundaries associated with each role. The safest default for this project is to use anonymised, fictional, or otherwise non-identifying examples wherever possible.

If there is any doubt about whether information is appropriate to enter into the prototype, it should be removed or anonymised first. If there is any doubt about whether cloud routing is permissible for a dataset, the data should not be sent until the relevant legal and organisational approvals are in place.

Privacy Policy

This privacy policy explains how Ambience-AI-1.5 may use and process data when the application is used for demonstrations, evaluation, or future deployment-oriented handover. Because the platform supports clinical consultation workflows, users should assume that anything entered into the system may require a higher standard of care than ordinary consumer application content.

What data do we collect?

Ambience-AI-1.5 may collect and process data that is necessary to run its role-based clinical workflow. This can include account and authentication data, role assignments, consultation titles, specialties, severity information, message content, specialist review notes, notifications, audit-log events, and uploaded reference or supporting documents.

In a real deployment, some of this information could include personal data or special category health data if users were to enter identifiable patient information. For this reason, demonstrations and academic use should avoid unnecessary real-world identifiers wherever possible.

How do we collect your data?

You provide data to the application directly when you register an account, log in, request password-reset or verification flows, create consultations, send clinical messages, upload files, review cases, or use admin functions. Additional system data is generated automatically as part of the platform workflow, including audit logs, status transitions, notification records, and service-level operational metadata.

How will we use your data?

Ambience-AI-1.5 uses collected data to authenticate users, enforce GP, specialist, and admin permissions, manage consultation workflows, generate and revise grounded AI responses, display notifications, support specialist review, and maintain an auditable record of key actions. Operational data may also be used for debugging, reliability monitoring, and testing.

How do we store your data?

Core application data is stored within the project database layer, based on PostgreSQL and pgvector. Some transient operational data may also pass through Redis-backed queues and caches as part of notifications, retries, and background processing. If the application is configured to use an external or cloud-hosted model provider, relevant prompt content may be transmitted to that configured endpoint as part of the AI generation workflow.

The safest assumption for this project is that data should be minimised and anonymised before use. This appendix does not claim that the current prototype provides a production-ready retention, deletion, or compliance framework.

What are your data protection rights?

Under UK GDPR, users may have rights that include access to their personal data, rectification of inaccurate data, erasure in certain circumstances, restriction of processing, objection to processing, and data portability. In a real deployment, these rights would need to be supported through clear operational and legal processes, not only through interface design.

Privacy policies of other websites and services

This website contains links to external websites and services, including GitHub and other third-party resources. If a deployment uses an external inference provider or links users to external platforms, those services will have their own privacy policies and terms. This privacy policy applies only to Ambience-AI-1.5 as described in this report website.

Changes to our privacy policy

This privacy policy should be reviewed whenever the project changes its hosting model, authentication flow, storage design, or inference routing path. In particular, any move from local development to a cloud-hosted clinical context would require a more formal and complete policy review.

Dependencies

This appendix records the main libraries, frameworks, and runtime services used by Ambience-AI-1.5. It is written as a readable dependency reference for the actual implemented stack rather than a raw package dump.

Frontend Application Dependencies

The user-facing client is a React single-page application for GP, specialist, and admin workflows. Its dependencies focus on routing, API access, dashboard rendering, and secure client-side session handling.

frontend-dependencies.md 
- react
- typescript
- vite
- react-router-dom
- axios
- tailwindcss
- recharts
- crypto-js

Together these packages provide the component model, typed frontend build setup, route structure, authenticated API communication, utility-first styling, dashboard charts, and client-side token protection used across the browser application.

Backend API Dependencies

The backend is built as a FastAPI service responsible for authentication, consultation lifecycle management, notifications, audit logging, and role-enforced clinical workflow control.

backend-dependencies.md 
- fastapi
- sqlalchemy
- alembic
- pydantic
- PyJWT
- bcrypt
- httpx
- redis-py

This backend stack covers the API layer, schema validation, database access, migrations, authentication, outbound service calls, and Redis-backed operational behaviour.

Retrieval and AI Dependencies

The RAG service is separated from the main backend and depends on document extraction, chunking, embedding, vector retrieval, and retry-oriented background processing. This is the part of the system that grounds responses against medical evidence rather than returning free-form outputs alone.

rag-dependencies.md 
- sentence-transformers
- torch
- pgvector
- PyMuPDF
- langchain-text-splitters
- rq
- httpx
- ollama
- Med42

These dependencies support embedding generation, vector search inside PostgreSQL, PDF extraction, document chunking, Redis-backed retry workers, and routing to either local Ollama-hosted inference or a cloud Med42 endpoint.

Infrastructure and Runtime Services

Ambience-AI-1.5 is designed as a multi-service deployment rather than a single process application. The supporting runtime stack handles persistence, queueing, reverse proxying, and development-time email flows.

infrastructure-dependencies.md

- PostgreSQL
- pgvector
- Redis
- nginx
- Mailpit

This runtime layer orchestrates the container stack, stores operational and vector data, supports cache and queue behaviour, handles reverse proxying, and provides local email testing.

Testing Dependencies
