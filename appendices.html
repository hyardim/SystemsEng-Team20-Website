This appendix page collects practical material that sits alongside the main technical report: user instructions, deployment instructions, GDPR-oriented handling notes, and publication placeholders for the development blog and monthly video deliverables.

User manual

GP workflow

  • Log in as a GP user.
  • Create a new consultation with title, specialty, and severity.
  • Send the first clinical question to trigger AI generation and auto-submission.
  • Open the consultation detail timeline to inspect the AI answer and its citations.
  • Monitor review status through the consultation state and notifications.

Specialist workflow

  • Open the submitted queue and self-assign a consultation that matches your specialty.
  • Inspect AI messages and citations in the detailed review view.
  • Approve, reject, request changes, or replace a specific AI answer manually.
  • Use review feedback to trigger a revised AI answer when needed.

Admin workflow

  • View the dashboard for consultation and AI activity statistics.
  • Inspect user records and role distributions.
  • Review chats globally when investigating system behaviour.
  • Open audit logs to trace operational and security-relevant events.

Deployment manual

Prepare the model runtime Ensure Ollama or a compatible external inference endpoint is available. The default local stack expects host access from Docker.
Confirm environment assumptions PostgreSQL credentials, database URLs, and RAG service URLs are defined in Compose and container environments.
Launch the stack Start the compose setup so frontend, backend, RAG service, and pgvector database are built and run together.
Verify health Confirm backend and RAG health endpoints respond, then log in via the frontend using seeded or registered users.
Ingest guideline data Ensure the RAG service has access to indexed guideline documents and vector rows before expecting grounded citations.

Deployment note

The repository also contains cloud-hosting notes for larger Med42 deployments and helper scripts under the serve directory. Those materials indicate planned or optional deployment paths beyond the default local Compose setup.

GDPR and privacy of data

Privacy posture in the current design

  • User accounts include identity and role metadata.
  • Consultations may store patient context as free-form JSON.
  • Audit logs record actions and timestamps for traceability.
  • Notifications and messages persist workflow history.

Implications

  • Any real deployment handling patient data would require clear lawful basis, minimisation, retention, and access-control policies.
  • Prompt payloads sent to external inference endpoints would require careful processor and transfer assessment.
  • Source documents and consultation data should be handled under strict storage and access governance.
  • This project should be described as an academic prototype, not a completed GDPR-ready clinical platform.

Honest statement

The repository demonstrates privacy-aware architectural concerns such as auditability and role separation, but it does not by itself constitute a full GDPR compliance pack.
