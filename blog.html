Weekly progression log for Ambience-AI-1.5

This blog follows a realistic week-by-week engineering progression. It is based on the requirements evidence trail and includes the original server setup period, the later infrastructure pivot, and an intentionally large testing phase at the end.

Timeline at a glance

The project moved through five practical phases: early discovery and requirement definition, original server setup, infrastructure pivot after access loss, feature completion and stabilization, then an extended final testing period.

The weekly entries below are the main source of detail and should be treated as the primary progression record.

Week 1-2: problem framing and first specialist interviews

We started by mapping the Advice and Guidance pain points with specialist input from Rheumatology and Neurology. The shared message was clear: faster responses are useful only when evidence is visible and clinical accountability is preserved.

During these first two weeks, we agreed that our target workflow should be a structured GP request that leads to a specialist-checked response rather than an unreviewed AI output. We also narrowed scope to neurology and rheumatology guidance so retrieval could be grounded in a realistic, manageable evidence base.

Week 3-4: requirement baseline and acceptance criteria

We converted interview findings into explicit functional and non-functional requirements, then prioritized them with MoSCoW. At this stage, we also identified key risk areas: uncertain model outputs, deployment uncertainty, and governance expectations.

We used this period to lock the must-have baseline around grounded retrieval, specialist approval and safe fallback behavior. In parallel, we drafted measurable acceptance criteria for latency, auditability and reliability, and planned the initial deployment path around Intel's Gaudi 2 server environment available at that time.

Week 5-6: original server setup and first integrations

A significant part of this period was spent setting up and validating the original Gaudi 2 server environment. We treated this as core engineering work, not overhead, because model runtime behavior and infrastructure constraints directly affected design.

A major milestone in this phase was installing the Med42 70B model on the server and getting a stable first run. That process took longer than expected because we had to work through low-level setup tasks first, including driver updates, runtime compatibility checks and environment configuration before inference would run reliably.

By the end of Week 6, we had environment checks, early retrieval and first-generation flow running in controlled scenarios. This gave us enough confidence to continue implementation on top of a live setup instead of only mock components.

Week 7-8: stabilization while still on the original server

We spent another two weeks improving robustness on the Gaudi 2 path and aligning implementation details with the agreed requirements before a major environmental change.

After the first Med42 70B runs succeeded, we focused on making that setup repeatable and less fragile. We cleaned up configuration steps, re-checked drivers after updates and documented the server setup decisions so the team could reproduce the working state.

Most of our effort here went into consistency and safety: we tightened citation mapping, refined specialist review behavior through feedback and documented deployment friction points. Those notes became important later when we had to rethink infrastructure.

Week 9-10: server access lost, architecture pivot starts

Around one month into infrastructure work, access to the original server was lost. This forced a planned rework: keep the clinical workflow intact while changing the deployment strategy to local-first processing with controlled cloud fallback.

This was the biggest disruption in the project timeline. Instead of dropping features, we reprioritized the order of work and redesigned provider routing so we could preserve safety and review guarantees while migrating away from the original server dependency.

Week 11-12: local-first route operational

By this point, the team had a working local-first path and resumed steady feature work. The key objective was to avoid user-facing workflow disruption despite backend routing and environment changes.

We re-established stable inference and retrieval behavior, then validated fallback behavior under failure scenarios. Importantly, we kept the consultation and specialist review journey consistent so the infrastructure pivot did not create a different user experience.

Week 13-14: workflow simplification and specialist gate hardening

Requirement refinement and prototype review led to a major product decision: merge assistant-like interactions into one consultation-based journey where generation, review and decision history stay connected.

This was an important simplification for us. We removed split interaction paths, enforced the specialist gate before GP-visible completion and improved traceability so review decisions were easier to follow and audit.

Week 15-16: platform breadth beyond core answering

We used this phase to complete the features that make the system operationally usable rather than a narrow demo.

During these weeks, we added the surrounding platform capabilities that clinicians and administrators would actually need in practice: structured submission, optional attachments, notification handling and admin-level observability.

Week 17-18: integration hardening and defect burn-down

Before entering final testing, we focused on reducing integration risk between frontend, backend and retrieval services.

We concentrated on edge cases in review-state transitions, reliability around retrieval and fallback routing, and a repeatable container-based setup. The goal was to enter the final testing stretch with fewer integration surprises.

Week 19-20: testing wave 1 (functional and integration)

The final stretch deliberately prioritized testing as a major body of work. We started with end-to-end functional checks and integration scenarios across role-based workflows.

We ran representative GP, specialist and admin scenarios end-to-end, including low-confidence refusal paths and fallback behavior. The defects found in this wave fed directly into an ongoing fix-and-retest loop.

Week 21-22: testing wave 2 (performance, reliability, usability)

This period focused on non-functional validation and repeated bug-fix loops. Testing was intentionally iterative so we could re-run critical scenarios after each fix batch.

We checked latency on routine cases, stress-tested failure and recovery paths, and refined accessibility and workflow clarity in core screens. Each fix was followed by another targeted run to confirm that we were improving without introducing regressions.

Week 23-24: testing wave 3, evidence packaging and handover readiness

Final weeks were dominated by validation evidence, final regressions and reporting consistency across requirements, implementation and testing documentation.

We finished with a final regression pass on high-priority workflows, consolidated test evidence for the report website and closed remaining defects. By the end of Week 24, we had a cleaner handover story that matched what we actually built and tested.

Closing note

This draft is intentionally realistic rather than perfectly linear. The largest hidden costs were infrastructure change and repeated testing cycles, both of which are now represented explicitly in the timeline.

Draft assumptions

This draft assumes about one month of original server setup before access loss and a deliberately extended final testing period. Exact calendar dates can be overlaid later.
