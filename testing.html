Evaluation and Testing

How the project was validated

Evaluation for this project is primarily engineering-driven. The codebase is accompanied by backend, frontend, RAG-service, and end-to-end tests that target workflow correctness, access control, notification behaviour, audit logging, and retrieval logic. This does not make the system clinically validated, but it does make the implementation inspectable and testable.

Backend testing focus

  • Authentication and profile management, including password reset and strength enforcement.
  • Chat lifecycle operations, title updates, submission behaviour, and access restrictions.
  • Specialist assignment, approval, rejection, and request-changes flows.
  • Notification creation, unread filters, and mark-as-read behaviour.
  • Admin stats and admin-only access control.
  • Security and RAG audit logging, including distinction between grounded answers and service failures.

Frontend testing focus

Auth and route state

Tests cover the auth context, protected routes, login, registration, and localStorage restoration behaviour.

Interaction components

Notification dropdowns, chat inputs, headers, badges, and message rendering are exercised in component tests.

Page-level flows

GP, specialist, and admin pages are tested with mocked API responses using MSW.

E2E smoke path

Playwright provides browser-level execution over a running stack.

RAG service testing focus

The RAG codebase has the widest test spread because it contains the most pipeline stages. Tests exist for ingestion, metadata extraction, chunking, embedding, vector store logic, retrieval filters, citation construction, reranking and routing helpers, prompt creation, and API routes.

Ingestion

Retrieval

Vector search, keyword search, filtering, fusion, reranking, citation logic, and query APIs.

Generation

Prompt construction, pipeline execution, routing rules, and API contracts.

Evaluation summary

Question Evidence in repository Interpretation
Does the workflow enforce roles? Protected frontend routes and role-specific backend dependencies. Yes, at both UI and API levels.
Are AI answers traceable? Messages store citation metadata; source PDFs can be opened via document endpoints. Partially yes; grounded answers are explicit when retrieval succeeds.
Is specialist oversight real? Assignment, review, request-changes, and manual-response flows are implemented in the specialist service. Yes, oversight is embedded in the business process.
Is the architecture production-hardened? Compose deployment, automated tests, and modular services exist, but some paths remain transitional. It is a serious engineering prototype rather than a fully hardened clinical product.

Limits of the evaluation

Important distinction

Automated tests validate software behaviour, not medical efficacy. The repository shows that the system behaves consistently under tested scenarios, but it does not by itself establish clinical safety or regulatory readiness.

  • No formal user study is represented in the current repository.
  • No benchmark suite for clinical answer quality is packaged alongside the app.
  • Model behaviour still depends on external runtime availability and the quality of ingested guideline sources.

Bottom line

The project is well tested as a software system. It should still be described as a specialist-reviewed consultation assistant, not as an autonomous diagnostic authority.
