Testing

A layered, evidence-based approach to quality

The project is validated through four complementary layers: unit tests that mirror the source folder structure of every service, integration tests that exercise full feature flows, Playwright end-to-end tests that drive the real browser across major user stories, and automated code-quality checks — linting, formatting, and type checking — enforced on every commit. All tests pass. Coverage is enforced in CI.

Testing strategy

Testing is structured as a pyramid. The base is wide unit coverage on individual functions and components. Integration tests sit above, verifying that features behave correctly when their parts work together. End-to-end tests at the top confirm complete user journeys through the live stack.

Layer Services covered Tooling What is verified
Unit Frontend, Backend, RAG service Vitest, Pytest Individual functions, components, and classes in isolation with mocked dependencies
Integration Frontend, Backend, RAG service Vitest + MSW, Pytest + SQLite / PostgreSQL Full feature flows — auth, ingestion, retrieval, chat lifecycle — with real databases
End-to-end Full stack Playwright Browser-driven user stories across GP, Specialist, and Admin roles
Code quality Frontend, Backend, RAG service ESLint, TypeScript, Ruff, MyPy Linting, formatting, and type correctness on every commit

Unit testing — Frontend

The frontend test suite uses Vitest and React Testing Library. Test files mirror the src/ folder structure exactly, so every module has a corresponding test file at the same path with a .test.ts(x) suffix.

Framework

Vitest with jsdom, React Testing Library, and @testing-library/user-event for realistic interaction simulation.

Coverage reporting

v8 provider with HTML, text, and LCOV reporters. Thresholds enforced for lines, branches, functions, and statements.

What is tested

Services

api.test.ts covers login, register, chat operations, notifications, specialist queue, and admin endpoints. api.streaming.test.ts covers streaming API behaviour.

Hooks

useChatStream.test.ts exercises the real-time streaming hook including connection state, message accumulation, and error handling.

Components

ChatMessage, ChatInput, NotificationDropdown, Header, ErrorBoundary, ProtectedRoute, and Badges are each tested in isolation with simulated props and user interactions.

Contexts

AuthContext is tested for login, logout, token persistence, and localStorage restoration so that auth state survives page refresh.

Pages

All six auth pages (login, register, forgot password, reset password, verify email, resend verification), GP pages, Specialist pages, and Admin pages each have dedicated test files.

Utilities

secureStorage.test.ts and messageMapping.test.ts verify encrypted storage helpers and message transformation logic.

Mocking strategy — MSW

All HTTP calls are intercepted by Mock Service Worker v2. The MSW server is started before tests and reset between each test to prevent state leakage. Handlers live in src/test/mocks/handlers.ts and cover every API endpoint with realistic fixture data — GP, Specialist, and Admin users, chats with messages, notifications, and audit logs. A renderWithProviders() utility wraps components with Router and AuthProvider, and a seedAuth() helper pre-populates encrypted localStorage to simulate a logged-in session without going through the login flow.

Unit testing — Backend

The backend suite uses Pytest with pytest-asyncio. 31 test files cover the full feature surface: authentication, chat lifecycle, specialist workflow, admin operations, caching, file upload, RAG integration, notifications, and security.

Framework

Pytest with pytest-asyncio for async route handlers, and FastAPI's TestClient for synchronous endpoint invocation.

Database isolation

A temporary file-based SQLite database is created per test session with tables built from the real SQLAlchemy models, shared between sync and async engines.

Feature areas covered

  • Authentication: registration, email verification flow, login, JWT issuance, password reset flow, and password strength enforcement.
  • Chat lifecycle: create, list, get, archive, send message, title updates, submission, and access restrictions between roles.
  • Streaming: streaming response handling and partial content delivery.
  • Specialist workflow: assignment, approval, rejection, request-changes, and queue operations.
  • Admin operations: user management, guideline CRUD, statistics endpoints, and admin-only access control.
  • Caching: 8 dedicated cache test files covering cache behaviour, disabled state, error paths, key generation, logging, patterns, payloads, and sync wrappers.
  • File upload and RAG: file upload, text extraction, patient context, RAG audit logging, and grounded-answer distinction.
  • Notifications: creation, unread filters, and mark-as-read behaviour.
  • Security: route protection, role enforcement, and token validation.

Fixture design

conftest.py provides a rich fixture hierarchy. User fixtures supply pre-registered GP, Specialist, and Admin accounts with their JWT authorization headers, so individual tests declare their role context declaratively. Chat fixtures provide chats at known lifecycle states (created, submitted). All fixtures use dependency injection overrides to swap the production database for the test SQLite engine, keeping tests hermetic.

Mocking approach

External dependencies — email sending, rate-limit state, feature flags — are patched with unittest.mock and monkeypatch. Auto-use fixtures clear rate-limit counters and reset email verification flags between every test, preventing inter-test contamination without manual cleanup code.

Unit testing — RAG service

The RAG service has the widest test spread: 47 test modules and approximately 10 500 lines of test code. Every stage of the pipeline — from document ingestion to final answer generation — has a corresponding test directory that mirrors the src/ structure.

Ingestion (14 files)

Text extraction, cleaning, chunking strategies, embedding generation, vector store writes, section detection, table detection, metadata extraction, web source sync, and the ingestion CLI.

Retrieval (10 files)

Vector similarity search, BM25 keyword search, result fusion, reranking, query expansion, metadata filtering, citation construction, and the retrieval CLI.

Generation (4 files)

Prompt construction with patient context, prompt construction with file context, query routing logic, and the LLM client interface.

RAG pipeline (3 files)

End-to-end pipeline execution, generation with retrieved context, and prompt assembly.

API routes (4 files)

RAG query endpoint, document ingestion route, guideline sync routes, and retry job management.

Infrastructure (6 files)

Async retry queue, background retry worker, streaming response handling, configuration management, database utilities, and logging.

Mocking strategy

Unit tests in the RAG service use unittest.mock.patch and MagicMock to isolate the component under test from its neighbours. make_*() factory helpers construct realistic test objects — chunks, embeddings, retrieval results — without needing a live database, so the bulk of the suite runs fast without any external services. Integration-style tests that exercise the vector store use a real PostgreSQL + pgvector instance (provided as a Docker service in CI).

Integration testing

Beyond isolated unit tests, each service contains tests that exercise complete feature flows. These tests use real (in-process) databases rather than mocks, verifying that the layers of the application work together correctly.

Authentication flow (backend)

Register → email verification token → activate account → login → JWT issuance → /me endpoint → password reset → password change. Tested as a chain against a real SQLite database.

Chat lifecycle (backend)

Create chat → send message → streaming response → specialist submission → specialist response → archive. The full GP-to-specialist consultation flow is exercised with real database writes.

Specialist queue flow (backend)

Chat submission → appearance in specialist queue → self-assignment → review → response or request-changes. Access control between GP and Specialist roles is verified at each step.

Admin operations (backend)

User listing, role changes, guideline CRUD, statistics aggregation, and audit log retrieval are tested end-to-end against a seeded database with multiple user types.

RAG ingestion flow (rag_service)

Document upload → text extraction → chunking → embedding generation → vector store write → retrieval verification. The pipeline is tested as a whole to catch cross-stage regressions.

RAG retrieval flow (rag_service)

Query → parallel vector + keyword search → score fusion → reranking → prompt construction → LLM generation. All stages are wired together in pipeline integration tests.

Database strategy per service

Backend integration tests use a temporary file-based SQLite database shared between sync and async engines, created from the production SQLAlchemy models. RAG service integration tests use a real PostgreSQL 16 + pgvector instance (provided as a Docker service container in GitHub Actions CI) to exercise vector operations realistically.

End-to-end testing — Playwright

Six end-to-end test suites in frontend/e2e/app.spec.ts drive a real browser against the running development stack. Tests execute on both Chromium and Firefox to catch browser-specific regressions.

Suite Scenarios covered
Authentication Login page loads correctly; invalid credentials show errors; valid GP credentials produce a successful login; navigation to registration page works.
Protected routes Unauthenticated users are redirected to login; the public landing page remains accessible without authentication.
GP workflow Consultation list loads; new consultation form opens and submits correctly; search filters results; profile navigation and logout complete the session.
Specialist workflow Specialist queries page loads; queue tab and assigned tab toggle correctly; query details are accessible.
Admin workflow User management page renders; chat oversight and audit log pages load; user search filters correctly.
Access control A GP-authenticated session cannot access admin-only pages; the application enforces role boundaries at the browser level.

CI configuration

Browsers

Chromium (Desktop) and Firefox (Desktop) — both configured in playwright.config.ts as named projects.

Retries and stability

2 retries in CI, 0 locally. Workers are sequential in CI and parallel locally. Traces are captured on the first retry for debugging flaky failures.

Dev server

Playwright auto-starts npm run dev before the test run, so no manual stack startup is needed in CI.

Reporting

HTML report generated after every run. Screenshots captured only on failure to keep artefact size small.

Code quality — linting, formatting, and type checking

All three services enforce code quality checks as part of the development workflow. These run in CI before tests, so a lint error or type violation blocks the pipeline before any test is executed.

Service Linter Formatter Type checker
Frontend ESLint with TypeScript-ESLint, React Hooks plugin, React Refresh plugin Prettier (via Vite build pipeline) TypeScript compiler (tsc -b) — strict mode, fails the build on type errors
Backend Ruff (rules: E, W, F, I, B, C4, UP) Ruff formatter (replaces Black; 88-char line length, double quotes) MyPy — disallow_untyped_defs = true, warn_return_any = true
RAG service Ruff (same rule set, Python 3.10+ target) Ruff formatter MyPy — same strict settings, run as mypy src in CI

Ruff rule categories

E / W — pycodestyle

Standard Python style errors and warnings. Catches whitespace issues, bare excepts, and statement hygiene.

F — Pyflakes

Unused imports, undefined names, and redefined variables. Catches dead code and typos in symbol names.

I — isort

Import ordering enforced automatically. All imports are sorted consistently across the codebase.

B — flake8-bugbear

Opinionated warnings about likely bugs — mutable default arguments, loop variable binding, and similar traps.

C4 — flake8-comprehensions

Encourages idiomatic list/dict/set comprehensions over verbose equivalent constructs.

UP — pyupgrade

Upgrades syntax to modern Python equivalents — Union[A, B] → A | B, f-strings over .format(), and so on.

CI pipeline and coverage enforcement

GitHub Actions runs the full quality pipeline on every push to main and on every pull request. The pipeline fails fast: quality checks run before tests, and a coverage shortfall fails the build.

RAG service CI pipeline (documented)

Step Command Fails pipeline if…
1. Ruff lint ruff check . Any lint rule violation is present
2. Ruff format check ruff format --check . Any file is not correctly formatted
3. MyPy type check mypy src Any type error is found in source files
4. Pytest with coverage pytest --cov=src --cov-report=term --cov-fail-under=90 Any test fails, or coverage drops below 90 %

PostgreSQL in CI

The RAG service CI spins up a PostgreSQL 16 container with the pgvector extension enabled as a service alongside the test runner. Health checks poll pg_isready every 5 seconds before tests start, so vector store tests always run against a real database rather than a stub.
