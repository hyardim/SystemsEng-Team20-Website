Testing strategy

Testing is structured as a pyramid. The base is wide unit coverage on individual functions and components. Integration tests sit above, verifying that features behave correctly when their parts work together. End-to-end tests at the top confirm complete user journeys through the live stack.

Unit testing — FrontendAll passing

The frontend test suite uses Vitest and React Testing Library. Test files mirror the src/ folder structure exactly, so every module has a corresponding test file at the same path with a .test.ts(x) suffix.

Framework

Vitest with jsdom, React Testing Library, and @testing-library/user-event for realistic interaction simulation.

Coverage reporting

v8 provider with HTML, text, and LCOV reporters. Thresholds enforced for lines, branches, functions, and statements.

api.test.ts covers the full API client surface: login, register, chat create/list/get, send message, notifications, specialist queue operations, and admin endpoints. api.streaming.test.ts covers streaming response handling and partial content delivery.

useChatStream.test.ts exercises the real-time streaming hook end-to-end: connection setup, message chunk accumulation, stream completion, and error handling when the connection drops or the server returns an error.

ChatMessage tests message rendering, attachment display, and guideline reference links. ChatInput tests submission, keyboard shortcuts, and disabled states. NotificationDropdown, Header, ErrorBoundary, ProtectedRoute, and Badges are each tested in isolation with simulated props and user interactions.

AuthContext.test.tsx verifies login, logout, token persistence, and localStorage restoration. Auth state is confirmed to survive page refresh — the context re-hydrates correctly from encrypted storage on mount.

All six authentication pages have dedicated test files: login, register, forgot password, reset password, verify email, and resend verification. Role-specific pages are also covered: GP queries and new query pages, the Specialist queries page, and the Admin users page. Each page test uses MSW to mock the API layer and React Testing Library to drive interactions.

secureStorage.test.ts verifies the encrypted localStorage helpers — write, read, and clear operations with correct key derivation. messageMapping.test.ts verifies message transformation logic used to normalise API responses before rendering.

Mocking strategy — MSW

All HTTP calls are intercepted by Mock Service Worker v2. The MSW server is started before tests and reset between each test to prevent state leakage. Handlers live in src/test/mocks/handlers.ts and cover every API endpoint with realistic fixture data — GP, Specialist, and Admin users, chats with messages, notifications, and audit logs. A renderWithProviders() utility wraps components with Router and AuthProvider, and a seedAuth() helper pre-populates encrypted localStorage to simulate a logged-in session without going through the login flow.

Unit testing — BackendAll passing

The backend suite uses Pytest with pytest-asyncio. Test files cover the full feature surface: authentication, chat lifecycle, specialist workflow, admin operations, caching, file upload, RAG integration, notifications, and security.

Framework

Pytest with pytest-asyncio for async route handlers, and FastAPI's TestClient for synchronous endpoint invocation.

Database isolation

A temporary file-based SQLite database is created per test session with tables built from the real SQLAlchemy models, shared between sync and async engines.

Registration, login, and /me endpoint. Email verification token generation, activation, and re-send. Password reset request, token validation, and completion. Password strength enforcement. Profile update operations.

Create, list, get, and archive chats. Send message and receive AI response. Title updates and submission behaviour. Access restrictions between roles — a GP cannot view another GP's chat, and a Specialist cannot access chats they are not assigned to. Streaming response handling and partial content delivery.

Specialist self-assignment to queued chats. Review approval, rejection, and request-changes flows. Specialist message sending. Queue visibility — only unassigned chats appear in the queue, and assigned chats move to the assigned list.

User listing, role changes, and account management. Guideline create, update, and delete. Statistics endpoint aggregation. Admin-only access control — non-admin users receive a 403 on every admin route.

Multiple dedicated test files cover the cache layer in depth: cache hit and miss behaviour, disabled-cache mode, error paths when Redis is unavailable, key generation correctness, structured logging events, cache patterns, payload serialisation, and sync wrapper behaviour around async cache calls.

File upload and storage. Text extraction from uploaded documents. Patient context injection. RAG audit logging — RAG_ANSWER records log chunk usage and citation count; RAG_ERROR records log the exception type. The grounded-response statistic only increments when citations are present, distinguishing grounded answers from fallback responses.

Notification creation on relevant events. Unread-only filter returns only unseen notifications. Mark-as-read updates state correctly. Notifications are scoped to the correct user — users cannot read each other's notifications.

Route protection — unauthenticated requests are rejected with 401. Role enforcement — requests with a valid token but the wrong role receive 403. Token validation — expired, malformed, and revoked tokens are all rejected correctly.

Fixture design

conftest.py provides a rich fixture hierarchy. User fixtures supply pre-registered GP, Specialist, and Admin accounts with their JWT authorization headers, so individual tests declare their role context declaratively. Chat fixtures provide chats at known lifecycle states (created, submitted). All fixtures use dependency injection overrides to swap the production database for the test SQLite engine, keeping tests hermetic.

Mocking approach

External dependencies — email sending, rate-limit state, feature flags — are patched with unittest.mock and monkeypatch. Auto-use fixtures clear rate-limit counters and reset email verification flags between every test, preventing inter-test contamination without manual cleanup code.

Unit testing — RAG serviceAll passing

The RAG service has the widest test spread, with the most pipeline stages to cover. Every stage of the pipeline — from document ingestion to final answer generation — has a corresponding test directory that mirrors the src/ structure.

The largest test directory covers every stage of the document ingestion pipeline: text extraction from PDFs and other formats, text cleaning and normalisation, chunking strategies and boundary behaviour, embedding generation, vector store writes, section detection, table detection and extraction, metadata extraction, web source ingestion and sync scheduling, and the ingestion CLI.

Vector similarity search and its scoring behaviour. BM25 keyword search. Score fusion combining vector and keyword results. Reranking of fused results. Query expansion strategies. Metadata filtering by source, date, and document type. Citation construction from retrieved chunks. The retrieval CLI.

Prompt construction with patient context injected — verifies that patient data appears in the correct position in the prompt. Prompt construction with uploaded file context. Query routing logic — tests that queries above and below the routing threshold are sent to the correct provider. LLM client interface and response parsing.

End-to-end pipeline execution wiring retrieval into generation. Generation with retrieved context — verifies that chunks are passed correctly and citations appear in the response. Prompt assembly — verifies the full prompt structure including system instructions, retrieved context blocks, and the user query.

The main RAG query endpoint — request validation, pipeline invocation, and response format. The document ingestion route — upload handling and pipeline trigger. Guideline sync routes — manual and scheduled sync trigger endpoints. Retry job management — job creation, status, and cancellation.

Async retry queue — job enqueue, dequeue, and failure handling. Background retry worker — processing loop, error recovery, and backoff behaviour. Streaming response handling — chunk delivery and connection teardown. Configuration management — environment variable parsing and defaults. Database utilities. Logger setup — handler registration, naming, and duplicate-handler prevention.

Mocking strategy

Unit tests in the RAG service use unittest.mock.patch and MagicMock to isolate the component under test from its neighbours. make_*() factory helpers construct realistic test objects — chunks, embeddings, retrieval results — without needing a live database, so the bulk of the suite runs fast without any external services. Integration-style tests that exercise the vector store use a real PostgreSQL + pgvector instance (provided as a Docker service in CI).

Integration testing

Beyond isolated unit tests, each service contains tests that exercise complete feature flows. These tests use real (in-process) databases rather than mocks, verifying that the layers of the application work together correctly.

Authentication flow (backend)

Register → email verification token → activate account → login → JWT issuance → /me endpoint → password reset → password change. Tested as a chain against a real SQLite database.

Chat lifecycle (backend)

Create chat → send message → streaming response → specialist submission → specialist response → archive. The full GP-to-specialist consultation flow is exercised with real database writes.

Specialist queue flow (backend)

Chat submission → appearance in specialist queue → self-assignment → review → response or request-changes. Access control between GP and Specialist roles is verified at each step.

Admin operations (backend)

User listing, role changes, guideline CRUD, statistics aggregation, and audit log retrieval are tested end-to-end against a seeded database with multiple user types.

RAG ingestion flow (rag_service)

Document upload → text extraction → chunking → embedding generation → vector store write → retrieval verification. The pipeline is tested as a whole to catch cross-stage regressions.

RAG retrieval flow (rag_service)

Query → parallel vector + keyword search → score fusion → reranking → prompt construction → LLM generation. All stages are wired together in pipeline integration tests.

Database strategy per service

Backend integration tests use a temporary file-based SQLite database shared between sync and async engines, created from the production SQLAlchemy models. RAG service integration tests use a real PostgreSQL 16 + pgvector instance (provided as a Docker service container in GitHub Actions CI) to exercise vector operations realistically.

Logging and observability

The system uses three complementary logging layers: a database-backed audit trail for user actions, structured cache operation logs, and a dual-handler file and console logger in the RAG service. All three are tested.

Database-backed audit trail (backend)

Every significant user action is written to an audit_logs database table with a user identifier, action type, optional detail string, and UTC timestamp. Audit records are queryable through the admin API with filters for action, category, user, and date range.

Category Actions logged
AUTH Login, logout, register, profile update, password reset request, password reset completion, email verification request, email verification sent, email verified
CHAT Create, view, update, delete, submit for review, auto-submit for review, AI response generated
SPECIALIST Assign specialist, review approve, review reject, review request-changes, specialist message
RAG RAG answer (grounded response with citations), RAG error (service unavailable or exception), RAG revise (specialist edits an AI answer)
ADMIN Admin-initiated user and guideline operations

RAG audit logging

RAG events are logged with enough detail to distinguish a grounded answer from a service failure. A RAG_ANSWER record stores the query length, number of top-k chunks requested, and how many chunks were actually used. A RAG_ERROR record stores the query length and the exception type. The rag_grounded_responses statistic only increments when citations are present, keeping grounded and fallback responses separately countable. test_rag_audit.py verifies all of this behaviour.

Structured cache logging (backend)

Events logged

cache.hit, cache.miss, cache.set, cache.delete, cache.disabled, cache.error — one event per cache operation.

Structured fields

key, user_id, resource, ttl, count, error — passed as extra so log processors can index them.

RAG service file and console logging

The RAG service configures a logger per module using a shared get_logger() utility. Each logger gets two handlers: a console handler at the configured log level (default INFO) and a file handler writing to logs/rag.log always at DEBUG level. Both the log level and log file path are configurable via environment variables. test_logger.py verifies handler setup, logger naming, and that repeated calls to get_logger() do not register duplicate handlers.

Frontend component logging

React components use scoped console calls with a [ComponentName] prefix — for example [ErrorBoundary], [Notifications], [GPQueryDetail] — so browser DevTools output is filterable by component without a dedicated logging library.

CI configuration

Browsers

Chromium (Desktop) and Firefox (Desktop) — both configured in playwright.config.ts as named projects.

Retries and stability

2 retries in CI, 0 locally. Workers are sequential in CI and parallel locally. Traces are captured on the first retry for debugging flaky failures.

Dev server

Playwright auto-starts npm run dev before the test run, so no manual stack startup is needed in CI.

Reporting

HTML report generated after every run. Screenshots captured only on failure to keep artefact size small.

Code quality — linting, formatting, and type checking

All three services enforce code quality checks as part of the development workflow. These run in CI before tests, so a lint error or type violation blocks the pipeline before any test is executed.

Service Linter Formatter Type checker
Frontend ESLint with TypeScript-ESLint, React Hooks plugin, React Refresh plugin Prettier (via Vite build pipeline) TypeScript compiler (tsc -b) — strict mode, fails the build on type errors
Backend Ruff (rules: E, W, F, I, B, C4, UP) Ruff formatter (replaces Black; 88-char line length, double quotes) MyPy — disallow_untyped_defs = true, warn_return_any = true
RAG service Ruff (same rule set, Python 3.10+ target) Ruff formatter MyPy — same strict settings, run as mypy src in CI

Ruff rule categories

E / W — pycodestyle

Standard Python style errors and warnings. Catches whitespace issues, bare excepts, and statement hygiene.

F — Pyflakes

Unused imports, undefined names, and redefined variables. Catches dead code and typos in symbol names.

I — isort

Import ordering enforced automatically. All imports are sorted consistently across the codebase.

B — flake8-bugbear

Opinionated warnings about likely bugs — mutable default arguments, loop variable binding, and similar traps.

C4 — flake8-comprehensions

Encourages idiomatic list/dict/set comprehensions over verbose equivalent constructs.

UP — pyupgrade

Upgrades syntax to modern Python equivalents — Union[A, B] → A | B, f-strings over .format(), and so on.

CI pipeline and coverage enforcement

GitHub Actions runs the full quality pipeline on every push to main and on every pull request. The pipeline fails fast: quality checks run before tests, and a coverage shortfall fails the build.

PostgreSQL in CI

The RAG service CI spins up a PostgreSQL 16 container with the pgvector extension enabled as a service alongside the test runner. Health checks poll pg_isready every 5 seconds before tests start, so vector store tests always run against a real database rather than a stub.
