System Design

Service boundaries, request flow, and review workflow

Ambience-AI-1.5 is designed as a full-stack, multi-service application. The frontend, backend, RAG service, and pgvector database each own a distinct part of the runtime. The result is a system that can separate consultation state management from evidence retrieval and model execution.

Deployment topology

Compose-orchestrated architecture

Frontend :3000 React 19 + TypeScript + Vite + Tailwind + React Router.
→
Backend :8000 FastAPI app exposing auth, chats, specialist, admin, notification, and RAG proxy routes.
RAG service :8001 FastAPI service for retrieval, grounded answer generation, revision, and PDF serving.
↔
PostgreSQL/pgvector :5432 Stores consultation data plus vectorised guideline chunks and metadata.
→
Model endpoint Ollama on the host machine or a compatible external provider.

Main request flow

Frontend submits a GP message A protected GP route calls the backend with the consultation identifier and message content.
Backend persists chat state The user message is stored immediately. If the chat was still open, it is auto-submitted for review.
Backend triggers RAG generation Query text, specialty, and severity are sent to the RAG service.
RAG service retrieves evidence The service embeds the question, runs vector search, filters unsuitable chunks, and builds a grounded prompt.
Answer and citations return The backend stores the AI response and citation metadata as a message in the consultation timeline.
Specialist review updates the workflow The consultation can move through assignment, active review, approval, rejection, or revision-request states.

Review workflow state machine

Consultation status progression

OPENChat exists but has not yet entered specialist workflow.
→
SUBMITTEDTriggered automatically after the first GP message or via explicit submit.
→
ASSIGNEDSpecialist self-assigns the case.
REVIEWINGSpecialist engages with the consultation, messages, or revision loop.
→
APPROVEDTerminal positive state.
or
REJECTEDTerminal negative state, potentially with feedback.

Message-level review overlays this chat-level state machine. Individual AI messages can carry their own review metadata, allowing specialists to reject or replace a single answer while keeping the broader consultation in progress.

Data model

Entity Purpose Selected fields
User Identity, role, specialty, and active-state management. Email, password hash, role, specialty, is_active.
Chat Consultation container and review state record. Title, status, specialty, severity, patient_context, specialist assignment, review metadata.
Message Conversation timeline including AI evidence. Sender, content, citations, is_generating, review_status, review_feedback.
Notification Workflow event delivery to users. Type, title, body, chat reference, read state.
AuditLog Security and workflow traceability. User id, action, details, timestamp.
FileAttachment Attachment metadata model for future upload workflows. Filename, path, type, size, uploader, chat link.

API responsibilities by service

Backend API

Authentication, registration, profile, chats, specialist review, notifications, admin metrics, and audit visibility.

RAG API

Health checks, retrieval query endpoint, grounded answer endpoint, revision endpoint, and document-serving endpoint.

Frontend router

Client-side view composition for landing pages, auth pages, role-protected task flows, and dashboards.

Architecture note

Backend startup currently relies on ORM create_all plus schema patch helpers, which is practical for coursework iteration but weaker than a strictly migration-only production workflow.

Ambience-AI-1.5

A technical systems report for a university software engineering project. The system under study is a role-aware clinical consultation platform that combines React, FastAPI, PostgreSQL with pgvector, and a grounded medical RAG service.

This website intentionally describes implementation details, deployment paths, safeguards, testing, and known limitations rather than product marketing claims.
