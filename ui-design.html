UI Design

Role-oriented interface design for consultation workflows

The frontend is not a single generic chat screen. It is a role-protected single-page application whose information architecture separates GP creation tasks, specialist review tasks, and admin observability. The interface design follows the workflow boundaries already present in the backend services.

Information architecture

Route-level structure

PublicLanding, login, registration, password reset.
→
GP routesConsultation list, new consultation form, consultation detail timeline.
Specialist routesQueue view, assigned view, review detail page.
→
Admin routesDashboard, users, chats, and audit logs.

The route tree in the frontend router demonstrates this separation explicitly. Role restrictions are enforced on the client through protected routes and reinforced at the backend API layer.

GP user experience

Consultation creation

The GP starts with a simple structured form: consultation title, specialty, severity, and question context.

Timeline detail view

Messages are shown chronologically with citations attached to AI outputs, helping the GP understand where evidence came from.

Status awareness

The UI surfaces whether a chat is open, submitted, assigned, reviewing, approved, or rejected.

Polling for progress

The detail page refreshes while AI generation or review activity is still in progress.

Specialist review experience

The specialist interface is designed around intervention rather than passive reading. It needs to support queue triage, self-assignment, message-level review, and manual correction of AI-generated content.

Queue and assigned views

Separate entry points reflect the distinction between newly submitted consultations and work already claimed by a specialist.

Per-message review

AI outputs can be approved, rejected, marked for change requests, or replaced with a manual clinical answer.

Revision loop visibility

Placeholder messages and polling make AI regeneration observable rather than opaque.

Admin observability experience

The admin dashboard uses charts to visualise status distributions, specialty splits, and daily AI query counts. This is not decorative; it reflects the admin service contract, which exposes active consultations, grounded-response counts, daily AI query data, and role-based user activity.

Charts

Bar, pie, and area charts are rendered using Recharts, matching the dashboard’s statistical API payload.

Operational drill-down

Separate admin pages provide user management, chat oversight, and audit-log exploration.

Interaction patterns and UX trade-offs

  • Auth state is stored in context and restored from localStorage so route protection survives refreshes.
  • Notifications are refreshed on a 15-second polling loop instead of websockets.
  • Deep-link navigation from notifications sends the user directly to the related consultation detail view.
  • Because this is a clinical workflow, interpretability beats novelty: messages, statuses, and citations are emphasised over conversational decoration.

UI limitation

The repository contains an older frontend API client that is no longer central to the active UI flow. This is a maintainability issue rather than a user-facing defect, but it is part of the implementation reality.

Design principle

Each role sees the minimum interface necessary for its responsibilities, reducing both cognitive load and accidental privilege overlap.
